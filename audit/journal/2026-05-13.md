### Mechanisms understood (Phase 3 entry)

- **Chronix's interaction layer is adapter-agnostic.** The session has no DOM bindings, no event listeners, no pointer-capture API calls ÔÇö adapters (Vue3/React) do that. The session exposes a pure-functional reducer trio (begin/advance/commit) over an immutable transaction value.

## CalendarRangeSelect transaction runtime (Phase 3 core complete)

### Reference reading (CalendarRangeSelect)

- No new upstream reading. Range-select is a fundamentally different interaction from drag/resize: there's no "original range" to displace; both ends come from the drag itself. Chronix's choice to work in time-space at this boundary (rather than pixel-space, like the other three kinds) is a chronix-native design call, not a port.

### Mechanisms understood (CalendarRangeSelect)

- **Time-space at the boundary, not pixel-space.** The caller has already converted the `pointerdown` pixel position to an `anchorTime: Date` via the axis. Reason: there's no anchor "original" to preserve a pixel offset against; the entire range is defined by the drag, so storing it directly as a `(anchorTime, currentTime)` Date-pair is more natural than pixel-then-convert.
- **Snap semantics diverge.** `BarDrag`/`BarResize` snap the **delta** (preserves the relationship to original range). `CalendarRangeSelect` snaps **each end independently** to a snap-grid multiple. The two semantics correspond to "shift this thing by a snap-grid amount" vs "land both ends on grid lines from scratch."
- **Snap may collapse to zero.** Both ends within ┬▒half-snap-duration of the same grid mark collapse to zero-duration. Caller's policy whether that's a valid event create.
- **Backward drag auto-swaps.** If user drags from a later time to an earlier time (`currentTime < anchorTime`), commit produces `{ start: min, end: max }` so the resolved range always has `start <= end` regardless of drag direction.

## select-to-create parity ÔÇö analyzed and intentionally tautological

Tried to add a recording-replay parity vitest for `CalendarRangeSelectTransaction`. Investigated three capture strategies; concluded a strict parity assertion would be **tautological** with parity tests already in the suite.

### Capture strategies investigated

1. **Auto-dismiss the prompt + read the new bar from DOM.** Failed: the upstream's `select` callback fires, prompt opens, but with `dialog.dismiss()` no event is added. No new bar to read.
2. **Accept the prompt with a sentinel title + read the new bar.** Verified the demo's `addEvent` DOES fire (sentinel title appears in the sidebar's `currentEvents` list ÔÇö `sentinelInDom=true`), but **the new bar never renders in day view** even after a 2s wait. Couldn't find an oracle to assert against.
3. **Intercept the upstream's `select` callback args.** Would require monkey-patching the scheduler options before mount via `page.addInitScript`, but the demo passes the callback directly into the Vue component ÔÇö there's no global window hook to intercept cleanly without modifying the demo (R2 read-only).

### Why a parity assertion would be tautological

The chronixÔćöupstream parity for `CalendarRangeSelectTransaction` reduces to:

- chronix: `resolvedRange = { start: anchorTime, end: currentTime }` (or swapped)
- upstream: `resolvedRange = { start: timeFromPx(pointerDown.x), end: timeFromPx(pointerUp.x) }`

Both sides compute time via `axisStart + (pointer.x - wrapperLeft) / pxPerMs`. The constituent pieces are already pixel-perfect-verified by other parity tests:

- **`pxPerMs` parity**: covered by the 6 slot-width parity tests (day/week/month/season/halfYear/year) ÔÇö chronix's `axis.slotWidth` matches the upstream's rendered slot spacing within 1px for every view.
- **`wrapperLeft` parity**: implicit in the bar-placement test, which uses chronix's `BarPlacementPass` output to predict bar pixel positions matching the upstream's rendered bars within 1px across 13 events. If `wrapperLeft` differed between chronix and the upstream, those 13 events would have systematic x offsets ÔÇö they don't.
- **`axisStart` parity**: implicit in the day-view tick-labels set-equality test (chronix's `axis.ticks[0].time` matches the upstream's leftmost tick), and again in the bar-placement test.

So given chronix and the upstream agree on `pxPerMs`, `wrapperLeft`, and `axisStart`, they MUST produce the same `resolvedRange` from the same pointer events. A recording-replay assertion would just re-test those three components, not exercise any new code path.

## PointerHitTester (Phase 3 hit-test layer)

### Reference reading (hit-test)

- None new this iteration. The reference's hit-test is woven through `HitDragging.queryHitForOffset` Ôćĺ `TimelineSlats.positionToHit` Ôćĺ `OffsetTracker` Ôćĺ `elementFromPoint`; reading them earlier (for the BarResize parity work above) was enough context for the chronix-side design, which is structurally simpler (pure-functional, no DOM probes).

### Mechanisms understood (hit-test)

- **Chronix hit-test stays in content space, not viewport space.** The function takes `contentX / contentY` ÔÇö viewport-relative coords plus the timeline body's scroll offset, measured from the same origin `PlacedBar.x` / `SwimlaneStrip.y` are. Adapters do the viewport-to-content math; the hit-test is pure data in, discriminated-union out.
- **Bar-zone partitioning lives inside the bar's geometry, not against a slot grid.** Three zones: `'bar-edge-start'` (leftmost `edgeZoneWidth` pixels), `'bar-edge-end'` (rightmost `edgeZoneWidth` pixels), `'bar-body'` (between). Edge default = 8 px, matching the upstream-reference's invisible resize-rect width. When the bar is narrower than `2 ├Ś edgeZoneWidth` the body zone disappears and the bar's center splits the two edges ÔÇö necessary because resize-edge thresholds collide on narrow bars.
- **Empty-row fallback is a separate code path, not a degenerate bar case.** When no bar matches but `contentY` lands inside some `SwimlaneStrip`, the result is `{ kind: 'empty-row', rowId }` ÔÇö what `CalendarRangeSelectTransaction` needs to start. The adapter converts `contentX` to a Date via the axis on its own.
- **Z-order is "later in `placedBars` array = on top".** No CSS z-index, no separate hit-priority field. `BarPlacementPass` already emits bars in a stable input order, so the same array drives both paint order and hit order.
- **Overlay groups pass through as opaque ids.** When a bar declares `BarSpec.pointerOverlayId` and that bar wins the hit, the result carries `overlayId`. The adapter applies that overlay's `PointerCaptureConfig` (e.g. `requireInitialHit: false` for the progress-handle path) ÔÇö chronix's interaction layer never resolves the id itself. This keeps `pointerOverlayId` working as the anti-regression hook #2 originally described (first-class IR field, not CSS afterthought) without making the hit-test pass care about overlay-group semantics.

### Chronix work derived (hit-test)

- `packages/gantt/src/interaction/pointer-hit-test.ts` ÔÇö `PointerHitTester` interface (single `test(input)` method) + `defaultPointerHitTester` implementation. ~75 lines of code; single pass over `placedBars` in reverse, single linear scan over `strips` for the empty-row fallback.
- `PointerHitTestInput` shape: `contentX`, `contentY`, `placedBars` (output of `BarPlacementPass`), `strips` (output of `RowSwimlaneLayout`), optional `overlayIdByBarId` (lookup map built from `BarSpec.pointerOverlayId`), optional `edgeZoneWidth` (default 8 px).
- `PointerHitResult` discriminated union: `'bar-body' | 'bar-edge-start' | 'bar-edge-end' | 'empty-row'`. Misses return `null`. Each bar-\* variant carries `barId` plus optional `overlayId`.
- 22 vitest in `pointer-hit-test.test.ts` covering: body / start-edge / end-edge / boundary cases (`x === bar.x` Ôćĺ start; `x === bar.x + width` Ôćĺ end); custom and zero `edgeZoneWidth`; narrow-bar edge-collision split at center; empty-row fallback; null on y above / below / in-gap; z-order (later wins on overlap); overlay-id pass-through; empty inputs; zero-width bars.

### Naming justifications (hit-test)

- `PointerHitTester` (vs e.g. `BarHitTester` or `ChartHitTester`): the responsibility is "given a pointer position, what zone does it map to?" ÔÇö the input drives the name, not the output. `Bar*` would have been wrong since the result kind includes `'empty-row'` which is not a bar.
- `PointerHitResult` discriminator values: full hyphenated kind names (`'bar-edge-start'` not just `'start-edge'`) ÔÇö adapters branch on these inside their pointerdown handler, where clarity beats brevity. Matches the existing `BarDragTransaction.kind = 'bar-drag'` shape from Phase 1.
- `contentX` / `contentY` (vs `x` / `y` or `position`): the prefix names the coordinate space explicitly. Hit-test bugs in the reference often stemmed from viewport-vs-content confusion; the API name closes that door at the call site.
- `overlayIdByBarId` (Map vs Record): the input bar set can be hundreds; a `Map` gets `O(1)` lookup without prototype-chain quirks, and it composes naturally with the IR transform `new Map(bars.filter(b => b.pointerOverlayId).map(b => [b.id, b.pointerOverlayId]))`.

### Open / parked (hit-test)

- **Progress-handle hit zones are not modeled as separate hit targets.** The upstream-reference renders the progress triangle in a separate SVG overlay group; chronix's hit-test currently only knows about bar bodies and bar edges. When the adapter MVP needs to start a `ProgressHandleTransaction`, the resolution path is: (1) bar declares `pointerOverlayId: 'progress-handle-group'`, (2) hit-test returns `{ kind: 'bar-body', overlayId: 'progress-handle-group' }`, (3) adapter checks `overlayId` and picks the right transaction kind. If the progress-triangle needs its own pixel-precise zone (not just "any pixel of the bar with progress"), introduce a `progressHandleZone?: { x, width }` field on `PlacedBar` and a `'progress-handle'` hit kind. Defer until an adapter forces it.
- **Hit-test does NOT compute time from `contentX`.** Empty-row hits return `rowId` only; the adapter calls `axisStart + (contentX / pxPerMs)` itself when it needs a Date for `CalendarRangeSelectTransaction.beginCalendarRangeSelect`. Bundling axis-aware time conversion into the hit-test would couple it to `PlannedAxis` and lose the no-axis-needed simplicity.
- **No `nearest-strip` snap when y lands in a row-spacing gap.** Strict containment, `null` for gaps. UX-wise some apps want "snap to nearest" (especially with 1-px gaps that feel like misses), but that's a layer above. Adapters can wrap the result and re-test with `┬▒1 y` if they want that behavior.

### Mechanisms understood (Phase 4.1)

- **Composable stays DOM-free.** `useGanttPointer` takes `(contentX, contentY)` numbers on `begin / advance`. The consumer owns `getBoundingClientRect` math + the `setPointerCapture` lifecycle. This keeps the composable testable without `happy-dom`'s spotty PointerEvent + bbox shims: tests call the three methods with numbers and verify the `onBar* / onSelect` callbacks fire with the right payloads.

- **`<ChronixGantt>` owns the DOM event handlers.** SVG `pointerdown` calls `setPointerCapture(pointerId)` only if the composable actually started a transaction ÔÇö prevents capturing pointers for clicks that landed outside any bar. `pointerup` releases capture and resets state. `pointermove` is a no-op when no transaction is active, so passive cursor motion has zero compute cost.

## Phase 4.3 ÔÇö axis-tick header band in `<ChronixGantt>`

### Mechanisms understood (Phase 4.3)

- **Bar group transform is the right shift mechanism ÔÇö not per-rect y offsets.** The bars' layout y comes from `BarPlacementPass` and is computed against `SwimlaneStrip.y` starting at 0. Adding the axis band by mutating every rect's y would force every test that asserts `rect.y` to learn about `headerHeight`. Instead wrap the bar group in a `<g transform="translate(0, headerHeight)">`: rect attributes stay identical to the layout output, and the visual offset is one parent-`<g>` away. Existing rect-attribute tests survive without changes; only the new "bar group transform" test references the offset directly.
- **Pointer math must subtract `headerHeight` symmetrically.** The hit-tester operates in bar-group content space, where strip y=0 means the top of the first row. If `getBoundingClientRect` returns `{top: T}` and the cursor is at `clientY = C`, contentY must be `C - T - headerHeight` to keep `bar.y` semantics intact. Failing to subtract makes a click on row 0 (content-y near 0) look like a click on row 0 plus `headerHeight` pixels ÔÇö usually still inside the strip, but bar-edge resize zones (8 px) flip from "start" to "end" silently.
- **Axis-row click filtering at the adapter, not the hit-tester.** Hit-tester input `contentY < 0` would never match a strip (strips have y Ôëą 0), so technically the hit-tester already filters axis clicks. But making the contract explicit ÔÇö early-return when `pos.y < 0` in `onPointerdown` ÔÇö documents intent and avoids relying on a "happens to return null" behavior of the hit-test layer. Filtering happens at `begin` only; `advance` / `commit` accept negative y because an in-flight drag should survive a brief cursor visit to the axis band without aborting.
- **`Intl.DateTimeFormat('zh-CN', { hour: 'numeric', hour12: false })` emits exactly `'0ŠŚÂ' ÔÇŽ '23ŠŚÂ'` in Node 22 + happy-dom + V8.** Verified empirically (`node -e "..."`). Not `'00ŠŚÂ'`.
