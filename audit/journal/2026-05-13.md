Both yield the same `maxLevel` for any input where no bars are dropped ‚Äî and the demo doesn't use max-stack capping, so the simpler algorithm is exact-equivalent for the parity dataset.
- **Boundary touches (`bar1.end === bar2.start`) share a level.** The upstream's hierarchy uses non-strict less-than for end ‚â§ start. Chronix mirrors via `levelEnds[l] <= bar.start`.
- **Out-of-axis bars don't count toward stacking.** k-ui's `slicer.sliceProps(...)` clips events to the active date range before placement; an event whose range falls entirely outside the rendered axis doesn't push its row taller.

### Chronix work derived (stacking)

- `packages/gantt/src/layout/bar-stack-height-pass.ts` ‚Äî `BarStackHeightPass` interface + `defaultBarStackHeightPass.compute`. Takes `{ bars, rows, axis, ...config }`, returns `{ heightsPerRow, heightByRowId }`. Defaults match the demo (barHeight 30, stackSpacing 10, topPadding 8, bottomPadding 4, minRowHeight = 38).
- 14 vitest cases covering: zero bars (min floor), explicit minRowHeight override, single in-range bar, stacking 2 / 3 / N bars, non-overlapping bars share level 0, boundary-touching bars share level, axis-range filter (before / after / spanning / out-of-range bars don't inflate), multi-row order preservation, orphan-rowId bars dropped silently.

## Phase 3 entry ‚Äî PointerCaptureSession + BarDrag

### Reference reading (Phase 3 entry)

- No new references this iteration. Phase 1 IR design already pre-baked the `requireInitialHit` flag on `PointerCaptureConfig` and the four transaction discriminants ‚Äî Phase 3 just adds the runtime that produces and mutates those values.

### Mechanisms understood (Phase 3 entry)

- **Chronix's interaction layer is adapter-agnostic.** The session has no DOM bindings, no event listeners, no pointer-capture API calls ‚Äî adapters (Vue3/React) do that. The session exposes a pure-functional reducer trio (begin/advance/commit) over an immutable transaction value.
- **`requireInitialHit` is the entire "separate-SVG-layer" escape valve.** Caller decides whether the pointerdown hit the subject element (hit testing is the caller's job ‚Äî DOM-specific) and passes a boolean; combined with the per-subject config, the session returns either a transaction or `null`. - **Advance is cumulative-from-origin, NOT cumulative-from-previous.** This makes the reducer idempotent over a pointer-move trace ‚Äî replay the same `currentPx` sequence and you get the same final transaction. The transaction carries `originPx` so advance is stateless.
- **Snap-to-grid rounds the time delta, not the pixel delta.** Pixel delta is converted to ms first; the snap is applied in ms. This avoids accumulating rounding error from view-zoom changes (snap at the time level, render at the pixel level).

## CalendarRangeSelect transaction runtime (Phase 3 core complete)

### Reference reading (CalendarRangeSelect)

- No new upstream reading. Range-select is a fundamentally different interaction from drag/resize: there's no "original range" to displace; both ends come from the drag itself. Chronix's choice to work in time-space at this boundary (rather than pixel-space, like the other three kinds) is a chronix-native design call, not a port.

### Mechanisms understood (CalendarRangeSelect)

- **Time-space at the boundary, not pixel-space.** The caller has already converted the `pointerdown` pixel position to an `anchorTime: Date` via the axis. Reason: there's no anchor "original" to preserve a pixel offset against; the entire range is defined by the drag, so storing it directly as a `(anchorTime, currentTime)` Date-pair is more natural than pixel-then-convert.
- **Snap semantics diverge.** `BarDrag`/`BarResize` snap the **delta** (preserves the relationship to original range). `CalendarRangeSelect` snaps **each end independently** to a snap-grid multiple. The two semantics correspond to "shift this thing by a snap-grid amount" vs "land both ends on grid lines from scratch."
- **Snap may collapse to zero.** Both ends within ¬±half-snap-duration of the same grid mark collapse to zero-duration. Caller's policy whether that's a valid event create.
- **Backward drag auto-swaps.** If user drags from a later time to an earlier time (`currentTime < anchorTime`), commit produces `{ start: min, end: max }` so the resolved range always has `start <= end` regardless of drag direction.

### Chronix work derived (CalendarRangeSelect)

- `pointer-capture-session.ts` ‚Äî `BeginCalendarRangeSelectInput`, `AdvanceCalendarRangeSelectInput`, `CommitCalendarRangeSelectInput/Output` types added; three new methods (`beginCalendarRangeSelect`, `advanceCalendarRangeSelect`, `commitCalendarRangeSelect`) on the session interface and impl. - 11 new vitest covering: begin (anchor pinning, requireInitialHit reject); advance (currentTime updates, anchor stays); commit forward / backward (auto-swap) / zero / snap rounds each end independently / snap collapse-to-zero / snap ‚â§ 0 treated as no-snap / rowId propagation; full begin/advance√óN/commit lifecycle.

### Naming justifications (CalendarRangeSelect)

- `anchorTime` / `currentTime` (vs `originTime` / `cursorTime`). "Anchor" emphasizes the role: this is the point the drag is pivoting around; "current" is where the cursor is right now. Matches the existing IR field names from Phase 1 type design.

## select-to-create parity ‚Äî analyzed and intentionally tautological

Tried to add a recording-replay parity vitest for `CalendarRangeSelectTransaction`. Investigated three capture strategies; concluded a strict parity assertion would be **tautological** with parity tests already in the suite.

### Capture strategies investigated

1. **Auto-dismiss the prompt + read the new bar from DOM.** Failed: the upstream's `select` callback fires, prompt opens, but with `dialog.dismiss()` no event is added. No new bar to read.
2. **Accept the prompt with a sentinel title + read the new bar.** Verified the demo's `addEvent` DOES fire (sentinel title appears in the sidebar's `currentEvents` list ‚Äî `sentinelInDom=true`), but **the new bar never renders in day view** even after a 2s wait. Probable cause: the upstream's `selectInfo.allDay` resolves to true (or some view-filter quirk) when the drag happens in a time-grid view, so the added event doesn't render in the visible time band. Couldn't find an oracle to assert against.
3. **Intercept the upstream's `select` callback args.** Would require monkey-patching the scheduler options before mount via `page.addInitScript`, but the demo passes the callback directly into the Vue component ‚Äî there's no global window hook to intercept cleanly without modifying the demo (R2 read-only).

### Why a parity assertion would be tautological

The chronix‚Üîupstream parity for `CalendarRangeSelectTransaction` reduces to:

- chronix: `resolvedRange = { start: anchorTime, end: currentTime }` (or swapped)
- upstream: `resolvedRange = { start: timeFromPx(pointerDown.x), end: timeFromPx(pointerUp.x) }`

Both sides compute time via `axisStart + (pointer.x - wrapperLeft) / pxPerMs`. The constituent pieces are already pixel-perfect-verified by other parity tests:

- **`pxPerMs` parity**: covered by the 6 slot-width parity tests (day/week/month/season/halfYear/year) ‚Äî chronix's `axis.slotWidth` matches the upstream's rendered slot spacing within 1px for every view.
- **`wrapperLeft` parity**: implicit in the bar-placement test, which uses chronix's `BarPlacementPass` output to predict bar pixel positions matching the upstream's rendered bars within 1px across 13 events. If `wrapperLeft` differed between chronix and the upstream, those 13 events would have systematic x offsets ‚Äî they don't.
- **`axisStart` parity**: implicit in the day-view tick-labels set-equality test (chronix's `axis.ticks[0].time` matches the upstream's leftmost tick), and again in the bar-placement test.

So given chronix and the upstream agree on `pxPerMs`, `wrapperLeft`, and `axisStart`, they MUST produce the same `resolvedRange` from the same pointer events. A recording-replay assertion would just re-test those three components, not exercise any new code path.

- `recordings/select-to-create/{before,mid,after}.png` + `log.json`: still useful as a smoke artifact (proves the demo's select-to-create flow works at the recorded time/view).
- The recording captures `wrapperLeft` alongside the pointer events, so a future parity vitest can be added cheaply if/when one becomes non-tautological (e.g. the `selectMirror` rectangle's bbox during drag ‚Äî actually a snap-time decision the upstream makes, distinct from the math chronix already shares).

### Mechanisms understood (BarResize parity)

- **Recording capture needed month-view, not day-view.** Day-view slot=60 makes bar widths ‚â• 1440px even for short events, so every bar's right edge sits past the visible body (`gantt-timeline-body-right` ‚âà 948px wide). Resize handles render at the bar edges; if an edge is outside the viewport, Playwright can't put the mouse on it. Month-view slot=65 with 31-day axis = 2015px wide axis still exceeds visible body, but several 4‚Äì10 day events fit entirely within visible bounds.
- **Resize handles need explicit `display: block` for headless Playwright.** The CSS rule `.gantt-event:hover .gantt-event-resizer { display: block }` doesn't reliably flip in headless Chromium for SVG groups ‚Äî observed empirically that `bar.hover()` + `page.waitForTimeout(80)` left the handle as `gantt-event-background` per `elementFromPoint`. Solution: inject `addStyleTag({ content: '.gantt-event-resizer { display: block !important; }' })` before the recording's pointer events. The CSS literal lives in `tooling/golden-runner/src/reference-dom-map.ts` as `RESIZER_ALWAYS_VISIBLE_CSS` (the allow-listed boundary file). Visibility-only override; resize zone geometry, hit targets, and `EventResizing` handlers are unchanged.
- **Snap gap: the upstream applies a non-1:1 pointer-to-bar multiplier on month-view resize, chronix doesn't model it.** Empirical probe at multiple drag distances on `event-resize-{left,right}` (constants applied to `pickAndResizeEdge.deltaPx`):

## PointerHitTester (Phase 3 hit-test layer)

### Reference reading (hit-test)

- None new this iteration. The reference's hit-test is woven through `HitDragging.queryHitForOffset` ‚Üí `TimelineSlats.positionToHit` ‚Üí `OffsetTracker` ‚Üí `elementFromPoint`; reading them earlier (for the BarResize parity work above) was enough context for the chronix-side design, which is structurally simpler (pure-functional, no DOM probes).

### Mechanisms understood (hit-test)

- **Chronix hit-test stays in content space, not viewport space.** The function takes `contentX / contentY` ‚Äî viewport-relative coords plus the timeline body's scroll offset, measured from the same origin `PlacedBar.x` / `SwimlaneStrip.y` are. Adapters do the viewport-to-content math; the hit-test is pure data in, discriminated-union out.
- **Bar-zone partitioning lives inside the bar's geometry, not against a slot grid.** Three zones: `'bar-edge-start'` (leftmost `edgeZoneWidth` pixels), `'bar-edge-end'` (rightmost `edgeZoneWidth` pixels), `'bar-body'` (between). Edge default = 8 px, matching the upstream-reference's invisible resize-rect width. When the bar is narrower than `2 √ó edgeZoneWidth` the body zone disappears and the bar's center splits the two edges ‚Äî necessary because resize-edge thresholds collide on narrow bars.
- **Empty-row fallback is a separate code path, not a degenerate bar case.** When no bar matches but `contentY` lands inside some `SwimlaneStrip`, the result is `{ kind: 'empty-row', rowId }` ‚Äî what `CalendarRangeSelectTransaction` needs to start. The adapter converts `contentX` to a Date via the axis on its own.
- **Z-order is "later in `placedBars` array = on top".** No CSS z-index, no separate hit-priority field. `BarPlacementPass` already emits bars in a stable input order, so the same array drives both paint order and hit order.
- **Overlay groups pass through as opaque ids.** When a bar declares `BarSpec.pointerOverlayId` and that bar wins the hit, the result carries `overlayId`. The adapter applies that overlay's `PointerCaptureConfig` (e.g. `requireInitialHit: false` for the progress-handle path) ‚Äî chronix's interaction layer never resolves the id itself. This keeps `pointerOverlayId` working as the anti-regression hook #2 originally described (first-class IR field, not CSS afterthought) without making the hit-test pass care about overlay-group semantics.

### Chronix work derived (hit-test)

- `packages/gantt/src/interaction/pointer-hit-test.ts` ‚Äî `PointerHitTester` interface (single `test(input)` method) + `defaultPointerHitTester` implementation. ~75 lines of code; single pass over `placedBars` in reverse, single linear scan over `strips` for the empty-row fallback.
- `PointerHitTestInput` shape: `contentX`, `contentY`, `placedBars` (output of `BarPlacementPass`), `strips` (output of `RowSwimlaneLayout`), optional `overlayIdByBarId` (lookup map built from `BarSpec.pointerOverlayId`), optional `edgeZoneWidth` (default 8 px).
- `PointerHitResult` discriminated union: `'bar-body' | 'bar-edge-start' | 'bar-edge-end' | 'empty-row'`. Misses return `null`. Each bar-* variant carries `barId` plus optional `overlayId`.
- 22 vitest in `pointer-hit-test.test.ts` covering: body / start-edge / end-edge / boundary cases (`x === bar.x` ‚Üí start; `x === bar.x + width` ‚Üí end); custom and zero `edgeZoneWidth`; narrow-bar edge-collision split at center; empty-row fallback; null on y above / below / in-gap; z-order (later wins on overlap); overlay-id pass-through; empty inputs; zero-width bars.

### Naming justifications (hit-test)

- `PointerHitTester` (vs e.g. `BarHitTester` or `ChartHitTester`): the responsibility is "given a pointer position, what zone does it map to?" ‚Äî the input drives the name, not the output. `Bar*` would have been wrong since the result kind includes `'empty-row'` which is not a bar.
- `PointerHitResult` discriminator values: full hyphenated kind names (`'bar-edge-start'` not just `'start-edge'`) ‚Äî adapters branch on these inside their pointerdown handler, where clarity beats brevity. Matches the existing `BarDragTransaction.kind = 'bar-drag'` shape from Phase 1.
- `contentX` / `contentY` (vs `x` / `y` or `position`): the prefix names the coordinate space explicitly. Hit-test bugs in the reference often stemmed from viewport-vs-content confusion; the API name closes that door at the call site.
- `overlayIdByBarId` (Map vs Record): the input bar set can be hundreds; a `Map` gets `O(1)` lookup without prototype-chain quirks, and it composes naturally with the IR transform `new Map(bars.filter(b => b.pointerOverlayId).map(b => [b.id, b.pointerOverlayId]))`.

### Open / parked (hit-test)

- **Progress-handle hit zones are not modeled as separate hit targets.** The upstream-reference renders the progress triangle in a separate SVG overlay group; chronix's hit-test currently only knows about bar bodies and bar edges. When the adapter MVP needs to start a `ProgressHandleTransaction`, the resolution path is: (1) bar declares `pointerOverlayId: 'progress-handle-group'`, (2) hit-test returns `{ kind: 'bar-body', overlayId: 'progress-handle-group' }`, (3) adapter checks `overlayId` and picks the right transaction kind. If the progress-triangle needs its own pixel-precise zone (not just "any pixel of the bar with progress"), introduce a `progressHandleZone?: { x, width }` field on `PlacedBar` and a `'progress-handle'` hit kind. Defer until an adapter forces it.
- **Hit-test does NOT compute time from `contentX`.** Empty-row hits return `rowId` only; the adapter calls `axisStart + (contentX / pxPerMs)` itself when it needs a Date for `CalendarRangeSelectTransaction.beginCalendarRangeSelect`. Bundling axis-aware time conversion into the hit-test would couple it to `PlannedAxis` and lose the no-axis-needed simplicity.
- **No `nearest-strip` snap when y lands in a row-spacing gap.** Strict containment, `null` for gaps. UX-wise some apps want "snap to nearest" (especially with 1-px gaps that feel like misses), but that's a layer above. Adapters can wrap the result and re-test with `¬±1 y` if they want that behavior.
