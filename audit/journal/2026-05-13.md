### Mechanisms understood (slot-width derivation)

- **k-ui doesn't hardcode per-view slot widths.** Slot width = `max(viewportWidth / slotCount, minLabelCell)`, where `minLabelCell = isTimeScale ? 4*13 : 5*13` (i.e. 52 for day/week clock-time labels, 65 for date labels like `15日三`). When the viewport is wide enough that one slot would naturally stretch past the floor, slots stretch; otherwise the floor holds and the timeline becomes scroll-wider than the viewport.
- **`isTimeScale` is per-view-type.** day/week → true (hour-based labels); month/season/halfYear/year → false (date-based labels). The chars-13 floor reflects fitting "≥4 / ≥5 Han chars + minor padding" without label truncation.
- **DOM probe confirms the formula at viewport 1440×900:** `.gantt-timeline-body-wrapper` width per view = day 1440, week 8736, month 2015, season 5980, halfYear 11960, year 23725 → divides cleanly by `slotCount` to give 60/52/65/65/65/65, which is exactly `max(1440/slotCount, floor)`. Empirically VIEWPORT.width drives the stretch term (matches k-ui's actualBodyWidth in this demo because the chart container is given full width).

## Stack-aware row height pass (BarStackHeightPass)

### Reference files read (stacking)

- `packages/gantt/src/resource-timeline/GanttView.tsx:2046-2148` (upstream reference) — `calculateRowHeights(rowNodes, eventMinHeight, eventSpacing, splitProps, tDateProfile, timelineWidth): number[]`. Computes per-row height via per-row slicing + `computeFgSegPlacements` + `firstEventTopPadding + maxBottom + 4`, floored at `minRowHeight = eventMinHeight + 8`.
- `packages/gantt/src/timeline/event-placement.ts:44-146` — `computeFgSegPlacements` returns `[placements, totalHeight]`; each placement has a `top` (level-coord, multiples of `height + spacing`). Stacking uses an interval-graph `SegHierarchy` for strict-order + max-stack capping.
- `packages/gantt/src/resource-timeline/GanttView.tsx:2348-2366` — `resolveFirstEventTopPadding(option, resourceId)` resolves a global number / per-resource map / default 8. Demo sets `resourceFirstEventTopPadding: 8` globally (DemoApp.vue:1576).

### Mechanisms understood (stacking)

- **The upstream reference uses an interval-graph stacking algorithm** for max-stack capping and strict-order. For chronix v1, simple greedy interval coloring is sufficient (assign each sort-by-start bar to the lowest level whose latest bar ended before this one starts). Both yield the same `maxLevel` for any input where no bars are dropped — and the demo doesn't use max-stack capping, so the simpler algorithm is exact-equivalent for the parity dataset.
- **Boundary touches (`bar1.end === bar2.start`) share a level.** The upstream's hierarchy uses non-strict less-than for end ≤ start. Chronix mirrors via `levelEnds[l] <= bar.start`.
- **Out-of-axis bars don't count toward stacking.** k-ui's `slicer.sliceProps(...)` clips events to the active date range before placement; an event whose range falls entirely outside the rendered axis doesn't push its row taller.

### Chronix work derived (stacking)

## Phase 3 entry — PointerCaptureSession + BarDrag

### Reference reading (Phase 3 entry)

- No new references this iteration. Phase 1 IR design already pre-baked the `requireInitialHit` flag on `PointerCaptureConfig` and the four transaction discriminants — Phase 3 just adds the runtime that produces and mutates those values.

### Mechanisms understood (Phase 3 entry)

- **Chronix's interaction layer is adapter-agnostic.** The session has no DOM bindings, no event listeners, no pointer-capture API calls — adapters (Vue3/React) do that. The session exposes a pure-functional reducer trio (begin/advance/commit) over an immutable transaction value.
- **`requireInitialHit` is the entire "separate-SVG-layer" escape valve.** Caller decides whether the pointerdown hit the subject element (hit testing is the caller's job — DOM-specific) and passes a boolean; combined with the per-subject config, the session returns either a transaction or `null`. - **Advance is cumulative-from-origin, NOT cumulative-from-previous.** This makes the reducer idempotent over a pointer-move trace — replay the same `currentPx` sequence and you get the same final transaction. The transaction carries `originPx` so advance is stateless.
- **Snap-to-grid rounds the time delta, not the pixel delta.** Pixel delta is converted to ms first; the snap is applied in ms. This avoids accumulating rounding error from view-zoom changes (snap at the time level, render at the pixel level).

## ProgressHandle transaction runtime (Phase 3)

### Reference reading (ProgressHandle)

- `packages/gantt/src/interaction/interactions/EventProgressing.ts:130-140, 207-209` — the upstream's progress-drag math:

  // begin (pointerdown):
  initialProgress = progress.value;
  initialX = mouseEvent.clientX;
  eventWidth = eventRect.width; // event seg's getBoundingClientRect

  // advance (dragmove):

## CalendarRangeSelect transaction runtime (Phase 3 core complete)

### Reference reading (CalendarRangeSelect)

- No new upstream reading. Range-select is a fundamentally different interaction from drag/resize: there's no "original range" to displace; both ends come from the drag itself. Chronix's choice to work in time-space at this boundary (rather than pixel-space, like the other three kinds) is a chronix-native design call, not a port.

### Mechanisms understood (CalendarRangeSelect)

- **Time-space at the boundary, not pixel-space.** The caller has already converted the `pointerdown` pixel position to an `anchorTime: Date` via the axis. Reason: there's no anchor "original" to preserve a pixel offset against; the entire range is defined by the drag, so storing it directly as a `(anchorTime, currentTime)` Date-pair is more natural than pixel-then-convert.
- **Snap semantics diverge.** `BarDrag`/`BarResize` snap the **delta** (preserves the relationship to original range). `CalendarRangeSelect` snaps **each end independently** to a snap-grid multiple. The two semantics correspond to "shift this thing by a snap-grid amount" vs "land both ends on grid lines from scratch."
- **Snap may collapse to zero.** Both ends within ±half-snap-duration of the same grid mark collapse to zero-duration. Caller's policy whether that's a valid event create.
- **Backward drag auto-swaps.** If user drags from a later time to an earlier time (`currentTime < anchorTime`), commit produces `{ start: min, end: max }` so the resolved range always has `start <= end` regardless of drag direction.

### Chronix work derived (CalendarRangeSelect)

- `pointer-capture-session.ts` — `BeginCalendarRangeSelectInput`, `AdvanceCalendarRangeSelectInput`, `CommitCalendarRangeSelectInput/Output` types added; three new methods (`beginCalendarRangeSelect`, `advanceCalendarRangeSelect`, `commitCalendarRangeSelect`) on the session interface and impl. The stale "v0 implements `BarDragTransaction` only" docstring updated to declare all four kinds implemented and to highlight the time-space-vs-pixel-space asymmetry of this last one.
- 11 new vitest covering: begin (anchor pinning, requireInitialHit reject); advance (currentTime updates, anchor stays); commit forward / backward (auto-swap) / zero / snap rounds each end independently / snap collapse-to-zero / snap ≤ 0 treated as no-snap / rowId propagation; full begin/advance×N/commit lifecycle.

### Naming justifications (CalendarRangeSelect)

- `anchorTime` / `currentTime` (vs `originTime` / `cursorTime`). "Anchor" emphasizes the role: this is the point the drag is pivoting around; "current" is where the cursor is right now. Matches the existing IR field names from Phase 1 type design.

## select-to-create parity — analyzed and intentionally tautological

Tried to add a recording-replay parity vitest for `CalendarRangeSelectTransaction`. Investigated three capture strategies; concluded a strict parity assertion would be **tautological** with parity tests already in the suite.

### Capture strategies investigated

1. **Auto-dismiss the prompt + read the new bar from DOM.** Failed: the upstream's `select` callback fires, prompt opens, but with `dialog.dismiss()` no event is added. No new bar to read.
2. **Accept the prompt with a sentinel title + read the new bar.** Verified the demo's `addEvent` DOES fire (sentinel title appears in the sidebar's `currentEvents` list — `sentinelInDom=true`), but **the new bar never renders in day view** even after a 2s wait. Probable cause: the upstream's `selectInfo.allDay` resolves to true (or some view-filter quirk) when the drag happens in a time-grid view, so the added event doesn't render in the visible time band. Couldn't find an oracle to assert against.
3. **Intercept the upstream's `select` callback args.** Would require monkey-patching the scheduler options before mount via `page.addInitScript`, but the demo passes the callback directly into the Vue component — there's no global window hook to intercept cleanly without modifying the demo (R2 read-only).

### Why a parity assertion would be tautological

The chronix↔upstream parity for `CalendarRangeSelectTransaction` reduces to:

- chronix: `resolvedRange = { start: anchorTime, end: currentTime }` (or swapped)
- upstream: `resolvedRange = { start: timeFromPx(pointerDown.x), end: timeFromPx(pointerUp.x) }`

Both sides compute time via `axisStart + (pointer.x - wrapperLeft) / pxPerMs`. The constituent pieces are already pixel-perfect-verified by other parity tests:

- **`pxPerMs` parity**: covered by the 6 slot-width parity tests (day/week/month/season/halfYear/year) — chronix's `axis.slotWidth` matches the upstream's rendered slot spacing within 1px for every view.
- **`wrapperLeft` parity**: implicit in the bar-placement test, which uses chronix's `BarPlacementPass` output to predict bar pixel positions matching the upstream's rendered bars within 1px across 13 events. If `wrapperLeft` differed between chronix and the upstream, those 13 events would have systematic x offsets — they don't.
- **`axisStart` parity**: implicit in the day-view tick-labels set-equality test (chronix's `axis.ticks[0].time` matches the upstream's leftmost tick), and again in the bar-placement test.

So given chronix and the upstream agree on `pxPerMs`, `wrapperLeft`, and `axisStart`, they MUST produce the same `resolvedRange` from the same pointer events. A recording-replay assertion would just re-test those three components, not exercise any new code path.

### What stays committed

- `recordings/select-to-create/{before,mid,after}.png` + `log.json`: still useful as a smoke artifact (proves the demo's select-to-create flow works at the recorded time/view).
- The recording captures `wrapperLeft` alongside the pointer events, so a future parity vitest can be added cheaply if/when one becomes non-tautological (e.g. the `selectMirror` rectangle's bbox during drag — actually a snap-time decision the upstream makes, distinct from the math chronix already shares).

- **Resize handles need explicit `display: block` for headless Playwright.** The CSS rule `.gantt-event:hover .gantt-event-resizer { display: block }` doesn't reliably flip in headless Chromium for SVG groups — observed empirically that `bar.hover()` + `page.waitForTimeout(80)` left the handle as `gantt-event-background` per `elementFromPoint`. Solution: inject `addStyleTag({ content: '.gantt-event-resizer { display: block !important; }' })` before the recording's pointer events. The CSS literal lives in `tooling/golden-runner/src/reference-dom-map.ts` as `RESIZER_ALWAYS_VISIBLE_CSS` (the allow-listed boundary file). Visibility-only override; resize zone geometry, hit targets, and `EventResizing` handlers are unchanged.
- **Snap gap: the upstream applies a non-1:1 pointer-to-bar multiplier on month-view resize, chronix doesn't model it.** Empirical probe at multiple drag distances on `event-resize-{left,right}` (constants applied to `pickAndResizeEdge.deltaPx`):

  | pointer Δ | LEFT bar Δ | RIGHT bar Δ | LEFT ratio | RIGHT ratio |
  | --------- | ---------- | ----------- | ---------- | ----------- |
  | ±30 px    | ∓65 (1d)   | ±65 (1d)    | 2.17×      | 2.17×       |
  | ±60 px    | ∓130 (2d)  | ±130 (2d)   | 2.17×      | 2.17×       |
  | ±65 px    | ∓130 (2d)  | ±130 (2d)   | 2.00×      | 2.00×       |
  | ±130 px   | ∓260 (4d)  | +325 (5d)   | 2.00×      | 2.50×       |

  Pattern looks like `round(pointer * 2 / slotWidth)` up to 65 px, then diverges for larger drags (LEFT stays 2×, RIGHT inflates to 2.5×). Auto-scroll ruled out (probed `gantt-timeline-body-{wrapper,right}` `scrollLeft` before and after — zero in every run). Source reading of `positionToHit` + `computeMutation` predicts 1× — none of the standard slot-index rules (floor / round / ceil) reproduce the observed 2× multiplier. The discrepancy is real but its mechanism is opaque to read-only inspection. Logged for future investigation if magnitude parity becomes a P0.

- **Chronix's `BarResizeTransaction` uses delta-snap (round `timeDeltaMs` to a multiple of `snapDurationMs`), not slot-hit snap.** Modeling the upstream's slot-hit snap would require chronix's commit to know the slot boundaries explicitly (Date array, not a uniform `snapDurationMs`). Chronix's current contract is simpler and adapter-agnostic; the parity test asserts the math chronix _does_ implement and documents the magnitude gap.

## PointerHitTester (Phase 3 hit-test layer)

### Reference reading (hit-test)

- None new this iteration. The reference's hit-test is woven through `HitDragging.queryHitForOffset` → `TimelineSlats.positionToHit` → `OffsetTracker` → `elementFromPoint`; reading them earlier (for the BarResize parity work above) was enough context for the chronix-side design, which is structurally simpler (pure-functional, no DOM probes).

### Mechanisms understood (hit-test)

- **Chronix hit-test stays in content space, not viewport space.** The function takes `contentX / contentY` — viewport-relative coords plus the timeline body's scroll offset, measured from the same origin `PlacedBar.x` / `SwimlaneStrip.y` are. Adapters do the viewport-to-content math; the hit-test is pure data in, discriminated-union out.
- **Bar-zone partitioning lives inside the bar's geometry, not against a slot grid.** Three zones: `'bar-edge-start'` (leftmost `edgeZoneWidth` pixels), `'bar-edge-end'` (rightmost `edgeZoneWidth` pixels), `'bar-body'` (between). Edge default = 8 px, matching the upstream-reference's invisible resize-rect width. When the bar is narrower than `2 × edgeZoneWidth` the body zone disappears and the bar's center splits the two edges — necessary because resize-edge thresholds collide on narrow bars.
- **Empty-row fallback is a separate code path, not a degenerate bar case.** When no bar matches but `contentY` lands inside some `SwimlaneStrip`, the result is `{ kind: 'empty-row', rowId }` — what `CalendarRangeSelectTransaction` needs to start. The adapter converts `contentX` to a Date via the axis on its own.
- **Z-order is "later in `placedBars` array = on top".** No CSS z-index, no separate hit-priority field. `BarPlacementPass` already emits bars in a stable input order, so the same array drives both paint order and hit order.
- **Overlay groups pass through as opaque ids.** When a bar declares `BarSpec.pointerOverlayId` and that bar wins the hit, the result carries `overlayId`. The adapter applies that overlay's `PointerCaptureConfig` (e.g. `requireInitialHit: false` for the progress-handle path) — chronix's interaction layer never resolves the id itself. This keeps `pointerOverlayId` working as the anti-regression hook #2 originally described (first-class IR field, not CSS afterthought) without making the hit-test pass care about overlay-group semantics.

### Chronix work derived (hit-test)

- `packages/gantt/src/interaction/pointer-hit-test.ts` — `PointerHitTester` interface (single `test(input)` method) + `defaultPointerHitTester` implementation. ~75 lines of code; single pass over `placedBars` in reverse, single linear scan over `strips` for the empty-row fallback.
- `PointerHitTestInput` shape: `contentX`, `contentY`, `placedBars` (output of `BarPlacementPass`), `strips` (output of `RowSwimlaneLayout`), optional `overlayIdByBarId` (lookup map built from `BarSpec.pointerOverlayId`), optional `edgeZoneWidth` (default 8 px).
- `PointerHitResult` discriminated union: `'bar-body' | 'bar-edge-start' | 'bar-edge-end' | 'empty-row'`. Misses return `null`. Each bar-\* variant carries `barId` plus optional `overlayId`.
- 22 vitest in `pointer-hit-test.test.ts` covering: body / start-edge / end-edge / boundary cases (`x === bar.x` → start; `x === bar.x + width` → end); custom and zero `edgeZoneWidth`; narrow-bar edge-collision split at center; empty-row fallback; null on y above / below / in-gap; z-order (later wins on overlap); overlay-id pass-through; empty inputs; zero-width bars.

### Naming justifications (hit-test)

- `PointerHitTester` (vs e.g. `BarHitTester` or `ChartHitTester`): the responsibility is "given a pointer position, what zone does it map to?" — the input drives the name, not the output. `Bar*` would have been wrong since the result kind includes `'empty-row'` which is not a bar.
- `PointerHitResult` discriminator values: full hyphenated kind names (`'bar-edge-start'` not just `'start-edge'`) — adapters branch on these inside their pointerdown handler, where clarity beats brevity. Matches the existing `BarDragTransaction.kind = 'bar-drag'` shape from Phase 1.
- `contentX` / `contentY` (vs `x` / `y` or `position`): the prefix names the coordinate space explicitly. Hit-test bugs in the reference often stemmed from viewport-vs-content confusion; the API name closes that door at the call site.
- `overlayIdByBarId` (Map vs Record): the input bar set can be hundreds; a `Map` gets `O(1)` lookup without prototype-chain quirks, and it composes naturally with the IR transform `new Map(bars.filter(b => b.pointerOverlayId).map(b => [b.id, b.pointerOverlayId]))`.

### Open / parked (hit-test)

- **Progress-handle hit zones are not modeled as separate hit targets.** The upstream-reference renders the progress triangle in a separate SVG overlay group; chronix's hit-test currently only knows about bar bodies and bar edges. When the adapter MVP needs to start a `ProgressHandleTransaction`, the resolution path is: (1) bar declares `pointerOverlayId: 'progress-handle-group'`, (2) hit-test returns `{ kind: 'bar-body', overlayId: 'progress-handle-group' }`, (3) adapter checks `overlayId` and picks the right transaction kind. If the progress-triangle needs its own pixel-precise zone (not just "any pixel of the bar with progress"), introduce a `progressHandleZone?: { x, width }` field on `PlacedBar` and a `'progress-handle'` hit kind. Defer until an adapter forces it.
- **Hit-test does NOT compute time from `contentX`.** Empty-row hits return `rowId` only; the adapter calls `axisStart + (contentX / pxPerMs)` itself when it needs a Date for `CalendarRangeSelectTransaction.beginCalendarRangeSelect`. Bundling axis-aware time conversion into the hit-test would couple it to `PlannedAxis` and lose the no-axis-needed simplicity.
- **No `nearest-strip` snap when y lands in a row-spacing gap.** Strict containment, `null` for gaps. UX-wise some apps want "snap to nearest" (especially with 1-px gaps that feel like misses), but that's a layer above. Adapters can wrap the result and re-test with `±1 y` if they want that behavior.

### Mechanisms understood (Phase 4 entry)

- **Adapters compose, they don't extend.** The `@chronixjs/gantt-vue3` package re-exports nothing the core doesn't already export — it wraps the four layout passes inside a `ComputedRef`-driven composable and renders the placed bars via a Vue3 component. Avoids forcing the call site into one of "always pass refs" or "always pass getters" — the Vue3 ecosystem has both conventions live.

## Phase 4.1 — useGanttPointer + interactive `<ChronixGantt>`

### Reference reading (Phase 4.1)

- None new. The interaction integration is pure composition over Phase 3's `defaultPointerHitTester` + `defaultPointerCaptureSession`.

### Mechanisms understood (Phase 4.1)

- **Composable stays DOM-free.** `useGanttPointer` takes `(contentX, contentY)` numbers on `begin / advance`. The consumer owns `getBoundingClientRect` math + the `setPointerCapture` lifecycle. This keeps the composable testable without `happy-dom`'s spotty PointerEvent + bbox shims: tests call the three methods with numbers and verify the `onBar* / onSelect` callbacks fire with the right payloads.
- **Hit-test result routes to one of three transactions.** `'bar-body'` + `editable=true` → `beginBarDrag`. `'bar-edge-{start,end}'` + `editable=true` → `beginBarResize` with the matching edge. `'empty-row'` + `selectable=true` → `beginCalendarRangeSelect` after the composable converts `contentX` to a Date via `axisStart + contentX / pxPerMs`. Other combinations leave `activeTransaction` `null`. The progress-handle path stays unwired — `PointerHitTester` currently doesn't model a `'progress-handle'` zone (parked in the Phase 3 hit-test journal).
- **`barRanges` is the consumer's hand-off for `originalRange`.** Commit needs each bar's source `TimeRange` to apply the transaction's pixel delta. The composable accepts `MaybeRefOrGetter<ReadonlyMap<barId, TimeRange>>`; for orphan barIds (in `placedBars` but missing from the map), commit is silently skipped — same orphan-safe contract as `BarPlacementPass`.
- **`snapDurationMs ≤ 0` is "no snap" by convention.** Both the chronix core's `commitBarDrag / commitBarResize` and `commitCalendarRangeSelect` already treat `≤ 0` as no-snap, so the SFC's default `snapDurationMs: 0` flows through verbatim. Avoids a `number | undefined` prop union that complicates Vue3 prop validation.
- **`<ChronixGantt>` owns the DOM event handlers.** SVG `pointerdown` calls `setPointerCapture(pointerId)` only if the composable actually started a transaction — prevents capturing pointers for clicks that landed outside any bar. `pointerup` releases capture and resets state. `pointermove` is a no-op when no transaction is active, so passive cursor motion has zero compute cost.
- **happy-dom's `getBoundingClientRect` returns `{0,0,0,0}` for un-laid-out SVG.** Convenient: in SFC tests, `clientX` equals `contentX` for free. Real-browser parity is unchanged because the composable doesn't depend on the bbox internally — only the SFC does, and the math `clientX - rect.left` reduces to identity when `rect.left = 0`.

### Naming justifications (Phase 4.1)

- `useGanttPointer` (vs `useGanttInteraction` / `useGanttDrag`): the responsibility is "wire pointer events to chronix interaction transactions", not "implement drag mechanics" — those live in `PointerCaptureSession`. The composable is a pointer-event-aware shim, and the name centers that role.
- `begin / advance / commit` (vs `down / move / up`): mirrors the chronix core's `beginBarDrag / advanceBarDrag / commitBarDrag` method triplet, so the call-site shape is consistent across layers. `down / move / up` would re-introduce DOM-event vocabulary at the composable boundary that the DOM-free design tries to avoid.
- `lastHit` exposed alongside `activeTransaction`: callers that want to render hover affordances (cursor changes, hit-zone highlights) need the hit-test outcome even when no transaction starts. `lastHit` makes the most recent `PointerHitResult` observable.

### Open / parked (Phase 4.1)

- **No `pointercancel` handling yet.** The browser can cancel a pointer (touch interrupted, focus stolen) mid-drag; the composable should reset `activeTransaction` on cancel without emitting a commit callback. Currently the only escape from a transaction is a pointerup. The SFC could listen on `pointercancel` and call `pointer.commit()` — but that fires an emit. Better: expose `pointer.abort()` that resets state without emitting. Defer until a touch-driven test or a real demo exposes the gap.
- **No progress-handle path.** Bars with `pointerOverlayId` get `overlayId` on their hit result, but the composable doesn't branch to `beginProgressHandle` because `PointerHitTester` doesn't return a `'progress-handle'` zone. The path is open: add `progress-handle` to `PointerHitResult` and a `useProgressHandle` config branch here.
- **No scroll plumbing.** The SFC assumes the SVG's content origin is at viewport `(rect.left, rect.top)` — no inner scroll container. When the example app lands with a real scroll-wrapper around the SVG, the SFC needs to query the wrapper's `scrollLeft / scrollTop` and add them to the pointer coords.
- **Pointer-event `wrapper.trigger` may not pass through `setPointerCapture`.** happy-dom + `@vue/test-utils` dispatch events synthetically; the `setPointerCapture` call inside the handler is a no-op in this environment. Tests still pass because pointermove → handler still fires inside the SVG bounds. Real-browser behavior diverges if a drag leaves the SVG, but the composable's `advance` re-enters cleanly on the next move.

## Phase 4.3 — axis-tick header band in `<ChronixGantt>`

### Reference reading (Phase 4.3)

- None new. Phase 2's `AxisRangePlanner` already emits `PlannedAxis.ticks: readonly AxisTick[]` with `{ x, time, label }`; the adapter just consumes it. The k-ui side's tick rendering lives in `packages/gantt/src/timeline/TimelineHeader.tsx` and is preact-specific — chronix doesn't port the render layer, only the planner output shape.

### Mechanisms understood (Phase 4.3)

- **Bar group transform is the right shift mechanism — not per-rect y offsets.** The bars' layout y comes from `BarPlacementPass` and is computed against `SwimlaneStrip.y` starting at 0. Adding the axis band by mutating every rect's y would force every test that asserts `rect.y` to learn about `headerHeight`. Instead wrap the bar group in a `<g transform="translate(0, headerHeight)">`: rect attributes stay identical to the layout output, and the visual offset is one parent-`<g>` away. Existing rect-attribute tests survive without changes; only the new "bar group transform" test references the offset directly.
- **Pointer math must subtract `headerHeight` symmetrically.** The hit-tester operates in bar-group content space, where strip y=0 means the top of the first row. If `getBoundingClientRect` returns `{top: T}` and the cursor is at `clientY = C`, contentY must be `C - T - headerHeight` to keep `bar.y` semantics intact. Failing to subtract makes a click on row 0 (content-y near 0) look like a click on row 0 plus `headerHeight` pixels — usually still inside the strip, but bar-edge resize zones (8 px) flip from "start" to "end" silently.
- **Axis-row click filtering at the adapter, not the hit-tester.** Hit-tester input `contentY < 0` would never match a strip (strips have y ≥ 0), so technically the hit-tester already filters axis clicks. But making the contract explicit — early-return when `pos.y < 0` in `onPointerdown` — documents intent and avoids relying on a "happens to return null" behavior of the hit-test layer. Filtering happens at `begin` only; `advance` / `commit` accept negative y because an in-flight drag should survive a brief cursor visit to the axis band without aborting.
- **`Intl.DateTimeFormat('zh-CN', { hour: 'numeric', hour12: false })` emits exactly `'0时' … '23时'` in Node 22 + happy-dom + V8.** Verified empirically (`node -e "..."`). Not `'00时'`. Not `'00:00'`. This is the literal we test against; if a future Node / ICU bumps the format, the tick-label vitest is the canary.
- **SVG height = `contentSize.height + headerHeight`.** Width is unchanged because the axis tick spacing is identical to bar layout x — they share `slotWidth`. The natural extension when headerRows lands is to add per-row heights to `headerHeight` (sum over `axis.headerRows.length`), not to introduce a new prop per row.

## Phase 4.6 — headerRows render + example view-toggle

### Mechanisms understood (Phase 4.6)

- **One `<rect>` + one `<text>` per header cell is enough.** Cell rendering is the simplest possible: a background-filled rect with a 1-px stroke as the outline, plus a centered `<text>` for `cell.label`. Adjacent cells share an edge; each rect's stroke draws over the shared boundary, giving a visually continuous separator without a separate `<line>` element. The doubled-stroke effect is invisible at 1px.
- **Default `headerRowHeight = 20`, opt-out via `0`.** Setting a non-zero default means the demo (and any naive consumer) gets the outer band for free in the day view — without it the band looks "incomplete." Tests that want the bare bar area pass `headerRowHeight: 0` explicitly; the headerRows render is then suppressed and the `<g class="cx-gantt-axis">` group sits flush at SVG `y=0`. The "0 hides" contract mirrors `headerHeight: 0`'s existing behavior.
- **`<g class="cx-gantt-axis">` gets its own transform now.** Previously it sat at SVG `y=0` because the tick row was the top band. With headerRows above it, the axis group is translated to `(0, headerRowsHeight)` so its child elements keep their `y ∈ [0, headerHeight]` coords. The bar group's transform sums both: `(0, headerRowsHeight + headerHeight)`. Pointer math subtracts the same sum, exposed as a `computed` so it stays in lockstep with `axis.headerRows.length × headerRowHeight + headerHeight`.
- **The reactive `axisInput` getter is what drives view-switching.** The `useGanttLayout` composable already accepted `MaybeRefOrGetter<AxisRangePlanInput>` on its input. Switching from a literal axisInput to a `computed(() => ({ ...viewId-driven... }))` re-derives the entire pipeline — `axis` → `strips` (BarStackHeightPass + swimlane) → `placedBars` — every time the user clicks a view button. No special "rebuild" or "invalidate" call is needed.
- **`overflow: auto` on the SVG wrapper is sufficient for horizontal scroll.** Week/month/season/halfYear/year views produce axes wider than the demo's chart pane (week: 52 × 168 = 8736 px; year: 65 × ~365 = ~23 700 px). Wrapping the SVG in a `overflow: auto` frame lets the browser handle the scroll bar; pointer math already uses `getBoundingClientRect().left`, which goes negative when the SVG scrolls left of the viewport, so `clientX - rect.left` still resolves to the correct content-x. No adapter change needed for horizontal scroll to "just work."
- **`Intl.DateTimeFormat('zh-CN', { weekday: 'short' })` emits `"周X"`, not `"X"`.** Chronix's planner uses `weekday: 'short'`, so month/season/halfYear/year tick labels look like `"1日周五"` — the reference DOM renders `"15日三"` (narrow weekday). Hidden divergence: the golden-runner parity tests don't compare tick labels for non-day views (only slot widths), so this was not caught. The Phase 4.6 SFC vitest covers what chronix actually produces and parks the divergence for a later parity-aligned planner change.

## Phase 3.bb — live-update for bar-drag + bar-resize

### Reference reading (Phase 3.bb)

- None new. The render-branch-on-`activeTransaction` pattern was established in Phase 3.z (progress-handle); 3.bb extends it to the other two pixel-space transactions.

### Mechanisms understood (Phase 3.bb)

- **`renderX / renderY / renderWidth` is the per-bar render frame computed once at the top of the bar's flatMap iteration.** All overlay nodes (fill, handle, label) read from the same frame, so they stay anchored to the bar's visible body during any in-flight transaction (drag or resize). - **Resize cross-over clamps at the render layer, not the math.** The math layer (`commitBarResize`) explicitly allows `start > end` because some chronix consumers may want to permit it (or reject with a custom message). The render layer is opinionated: a negative width would render nothing useful (SVG quietly drops negative-width rects), so we `Math.max(0, ...)` and let the user see the bar collapse — clear "you've overshot" signal that's still recoverable by pulling back.
- **`'barId' in activeTxn`** narrows the union from `AnyTransaction` to the three transaction kinds that carry a `barId` (`bar-drag`, `bar-resize`, `progress-handle`), excluding `calendar-range-select` (which has `rowId`). TypeScript's `in` operator narrows nicely; no explicit type predicate function needed.
- **bar-drag y-delta is tracked but its visual impact is minimal in chronix v1.** The bar's `y` only shifts during the drag preview; on commit the demo writes the new range (which doesn't include y) and the bar's row is unchanged. To actually change rows on drag, the commit path would need to map `deltaY` to a row jump — Park 4.x candidate (see use-gantt-pointer's roadmap).

## Phase 3.cc — `pointer.abort()` + pointercancel handler

### Reference reading (Phase 3.cc)

- None new. The need for an abort path was parked under Phase 4.1's "no `pointercancel` handling yet" item; everything was already wired except for the explicit abort method and the cancel handler.

### Mechanisms understood (Phase 3.cc)

- **`abort()` is `commit()` minus the emit.** The state-reset half of `commit` (clear `transaction.value`, clear `lastHitResult.value`) is the only piece needed. Refactoring commit to share the reset would be over-engineering — two lines duplicated is cheaper than threading a "should-fire-callback" flag.
- **`pointercancel` is the right DOM event to listen for.** Browsers fire it when a pointer interaction is interrupted: touch cancellation by the OS, focus stolen by another window, certain gestures. Listening on `pointercancel` mirrors how native scrollers / dragger libraries handle interruption — it's the canonical signal that "the user's intent is no longer reachable."
- **Pointer capture release happens in the cancel handler too.** Same as `commit` — once the transaction's lifecycle ends, the SVG should stop receiving captured pointer events. The `releasePointerCapture` call is idempotent and safe to issue even if the browser already released capture as part of the cancellation.
- **No emit fires on cancel.** This is the key semantic: `commit` says "the drag finished with a meaningful intent"; `abort` says "the drag never reached commit, drop the state." Demos / consumers reading the events should NEVER see a synthetic `bar-drop / bar-resize / bar-progress / select` payload from a cancellation — that would silently corrupt their state.
- **`abort()` when idle is a safe no-op.** Both refs are already null; assigning null is harmless. The early-return in `onPointercancel` (when no transaction is active) avoids the unnecessary `releasePointerCapture` call but `abort()` itself doesn't need an internal guard.

### Chronix work derived (Phase 3.cc)

- `adapters/vue3/src/use-gantt-pointer.ts`: added `abort()` to `UseGanttPointerOutput`. Two-line implementation: `transaction.value = null; lastHitResult.value = null`. JSDoc explicitly states no commit callback fires — frames the contrast against `commit()` for readers.
- `adapters/vue3/src/chronix-gantt.ts`: new `onPointercancel` handler. Mirrors `onPointerup`'s structure (early-return when idle, capture release) but calls `pointer.abort()` instead of `pointer.commit()`. Wired into the SVG root via the `onPointercancel` event binding.
- 4 new composable vitest covering: abort clears in-flight without emit, abort then advance is no-op, abort then begin starts fresh, abort while idle is safe.
- 4 new SFC vitest covering: pointercancel mid-bar-drag, mid-bar-resize, mid-progress-handle, and pointercancel while idle.

### Naming justifications (Phase 3.cc)

- `abort()` (vs `cancel()` / `reset()`): "abort" matches the semantic of "the transaction did not complete its intended action" (cf. `AbortController` in the web platform). "cancel" overloads with `pointercancel` (the DOM event triggering it) and would read awkwardly as both event name and method name. "reset" suggests returning to a clean state with no implication about whether intent was lost.
- `onPointercancel` (no separator) follows Vue's `on<NativeEvent>` convention (matching `onPointerdown / onPointermove / onPointerup`); kebab-case `on-pointercancel` would be the template-prop variant, but in `h()` render functions Vue accepts both.

### Open / parked (Phase 3.cc)

- **No `pointerleave` handling.** When the user drags off the SVG bounds, modern browsers still fire `pointermove / pointerup` because of `setPointerCapture`. But on older platforms (and certain touch flows) the pointer can "leave" without firing capture-aware events. A future hardening could listen on `pointerleave` and abort if no pointer is captured. Defer until a real-world regression surfaces.
- **No keyboard escape.** Pressing Escape during a drag is the convention in many drag-and-drop libraries. Adding a global Escape listener would require either window-level binding (lifecycle-aware) or a focus-aware element. Skipped to keep the SFC DOM-free of focus management.
- **`abort()` from the consumer's `bar-*` handler.** A demo might want to validate the commit value and abort if invalid (e.g. dropping a bar outside its allowed row). Currently the commit callback fires synchronously inside `pointer.commit()`; there's no way to "un-commit" after the emit. A future `pointer.commitWithValidation((payload) => boolean)` hook could thread this through, but the existing pattern (demo validates and silently writes back the unchanged value) already works.

## Phase 3.cc — DONE

- The composable exposes `abort()`; `<ChronixGantt>` listens for `pointercancel` and calls it. In-flight transactions cleared by browser-initiated cancellation no longer fire stray commit callbacks.
- **259/259 vitest cumulative** (was 251; Phase 3.cc added 8 tests — 4 composable + 4 SFC). ci-check fully green.

## Phase 4.5 — sticky-header scroll wrapper

### Reference reading (Phase 4.5)

- Re-read the reference demo's split-pane layout (`.gantt-timeline-layout` → `.gantt-timeline-header-{left,right}` over `.gantt-timeline-body-{left,right}`), the wrapper hierarchy inside the right pane (`.gantt-timeline-header-wrapper` / `.gantt-timeline-body-wrapper` carry the full natural-width content; ancestor panes clip via `overflow: hidden / auto`), and how the reference syncs horizontal scroll between header-right and body-right panes (shared horizontal scroll, independent vertical scroll on each).
- Re-read the design note at `audit/PHASE_4_5_DESIGN.md` (Z-approved). Three candidates were considered: X (two SVGs, JS scroll-sync), Y (single SVG with `<foreignObject>` + CSS sticky inside), Z (overflow:auto wrapper, SVG body, header SVG with `position: sticky`). Z chosen for zero-JS sync, broad browser support, and natural future extension to a 3-pane layout when a resource panel lands.

### Mechanisms understood (Phase 4.5)

- **`position: sticky` requires a scrollport ancestor.** The header SVG sticks relative to the nearest scroll container (the wrapper div with `overflow: auto`). If the wrapper's overflow stays `visible` (the default), sticky promotes to a more distant scroll container (e.g. `body` or `.cx-demo-svg-frame`), which collapses the intended pinning. Library inlines `overflow: auto` on the wrapper to guarantee a scrollport without requiring consumers to set it.
- **Sticky has no visible effect when the scrollport never actually scrolls.** With `overflow: auto` but no height cap, the wrapper grows to fit content and no scroll engages — sticky degrades to `position: static` semantically (header still flows in normal position). The consumer's CSS controls when sticky activates (e.g. demo sets `.cx-gantt-wrapper { max-height: 70vh }`). Document this in the wrapper's inline comment so a future maintainer doesn't add height to the library default.
- **`width: max-content` for screenshot capture, NOT for production.** The wrapper's `display: block` default makes its bounding box clamp to the parent's content-area width (= viewport width). A wider-than-viewport body SVG paints past the wrapper's box, but `locator.screenshot()` captures only the bbox — so a year-view (~24000 px) capture would miss everything past viewport-width. Solution: the VRT spec's hide-chrome CSS adds `width: max-content !important` on the wrapper so its bbox hugs the natural content width. The library does NOT ship `width: max-content`; the production wrapper stays block-sized so consumers can lay it out with regular CSS techniques.
- **Header SVG must occlude the body during scroll.** With `position: sticky; top: 0` the header floats over the body as the user scrolls vertically — without an opaque `background` on the header, bars peek through. Inline `background: #ffffff; z-index: 1` settles this. `z-index: 1` only matters in the (rare) case the body SVG is itself positioned and would otherwise stack equal-or-above the header.
- **happy-dom's `getBoundingClientRect()` returns `{0, 0, 0, 0}` for un-laid-out SVGs.** Test code reads pointer coords as `e.clientX - rect.left` → `e.clientX - 0` → `clientX`. With the old single-SVG root, the test had to add `headerBandHeight (44)` to clientY to land inside the bar area (because content-y 0 was 44 below SVG top). After splitting into separate SVGs, the body SVG's origin IS content-y 0 — `cy = 20` directly hits row 1. Updated ~30 SFC tests to drop the `+ 44` offset.
- **Vue's `style` prop accepts objects; `zIndex` camelCase auto-serializes to `z-index`.** Used inline-style objects throughout (`style: { position: 'sticky', top: '0', zIndex: 1, background: '#ffffff' }`) for clarity. The serialized DOM gets `style="position: sticky; top: 0px; z-index: 1; background: rgb(255, 255, 255);"` — happy-dom's CSSOM accepts `el.style.position === 'sticky'` for assertions.

### Chronix work derived (Phase 4.5)

### Mechanisms understood (Phase 5)

- **`position: sticky` in two dimensions is a one-property generalization of Phase 4.5's sticky-top.** With the wrapper as the scrollport, `sticky; top: 0; left: 0` on the top-left corner element pins it under any combination of horizontal + vertical scroll. The browser's compositor solves the sync; no JS handler is needed. The reference's JS scroll-sync between four panes (header-right ⇄ body-right, body-left ⇄ body-right) is a pre-`position:sticky` workaround.
- **CSS grid with `grid-template-columns: ${sidebarWidth}px auto` makes the wrapper a 2-track container that grows to fit the chart.** The `auto` track sizes to its content's intrinsic width (`max-content` of the body SVG = the chart's natural pixel width). The grid's overall intrinsic width = sidebarWidth + chart natural width. The wrapper's existing `overflow: auto` then engages horizontal scroll exactly as it did pre-sidebar. Using `1fr` instead of `auto` would fail: `1fr` resolves to "available space" so the grid's width never exceeds the wrapper's own width, no horizontal overflow detected, no scroll.
- **Z-index ladder ensures correct paint stacking at corner intersections.** When both axes scroll, the four panes geometrically overlap at the corners: top-left corner of every pane lands on top of the corner just before it. Without z-index, the body SVG (last in source order) would paint OVER the sidebar-body at the left-edge crossing and OVER the chart-header at the top-edge crossing. Monotone ladder (`sidebar-header: 3 > chart-header: 2 > sidebar-body: 1 > body: auto`) makes the correct pane win at each crossing. The opaque `background: #ffffff` on each non-default pane ensures the painting actually occludes (transparency would defeat the ladder).
- **`width: max-content` on a `display: grid` container resolves to the sum of intrinsic track widths.** For the VRT capture (which needs the wrapper's bbox to hug natural content for full-width screenshots), `width: max-content !important` on the grid wrapper produces `sidebarWidth + max-content of chart` — exactly the same result the natural overflow would produce, just promoted to the wrapper's own bbox.
- **happy-dom serializes Vue inline style objects predictably.** `style: { position: 'sticky', top: '0', left: '0', zIndex: 3 }` → `el.style.position === 'sticky'`, `el.style.top === '0px'`, `el.style.left === '0px'`, `el.style.zIndex === '3'`. The `zIndex` camelCase auto-serializes to `z-index` CSS property; the value is stringified ("3" not 3). Tests read via `Number(el.style.zIndex)` to compare numerically.

## Phase 5.x — vGrouping rowspan merge (2026-05-15)

### Reference reading (Phase 5.x)

- `examples/gantt/vue3/src/DemoApp.vue` lines 1273–1290 — the demo's `resourceAreaColumns` config: 3 columns (`airportName`, `baseName`, `workshopName`). ### Mechanisms understood (Phase 5.x)

- **Adjacency, not equality.** The merge rule is "adjacent + equal" — two rows with the same value separated by a different-valued row don't merge. A sandwich pattern (`海口`, `三亚`, `海口`) produces three rowspan=1 cells, not two. This matches the reference's behavior and keeps the render deterministic when rows reorder.
- **rowSpan height must match the body's strip y-range, not just the row count.** A `rowspan=N` `<td>` visually spans N consecutive `<tr>` rows. For the cell to align with the body's strips at the right y-range, each `<tr>` height must equal `strip.height + rowSpacing` for non-last rows (and `strip.height` for the last). Baking `rowSpacing` into the `<tr>` heights makes the table's total height equal the body's content height, and any `rowspan=N` cell automatically spans `N × strip.height + (N-1) × rowSpacing` (the body's y-range for those strips + gaps).
- **HTML `<table>` is the right tool for rowspan, even mid-CSS-grid.** CSS grid's `grid-row: span N` doesn't merge cell content — only the track allocation. For visual merge (one cell painted across multiple rows), HTML `<td rowspan="N">` is the native primitive. The wrapper's outer `display: grid` (for sidebar / chart panes) and the sidebar's inner `<table>` coexist cleanly — the table sits inside one grid cell.
- **`<colgroup>` shares per-column widths between header and body.** Both the sidebar-header `<table>` and the sidebar-body `<table>` carry a `<colgroup>` derived from the same `ColumnSpec.width` values. CSS `table-layout: fixed` then resolves column widths from `<col>` widths, so per-column vertical borders align across the two tables by construction. No JS layout sync needed.
- **`border-collapse: collapse` is required for rowSpan visual continuity.** With `border-collapse: separate`, each row's cell borders are drawn independently; a rowspan cell would show internal cell-border lines at the row boundaries. `collapse` merges adjacent borders into one — the rowspan cell renders as a single tall block with one continuous outer border.
- **Vue's `h()` accepts both `rowspan: N` and `rowSpan: N` for HTML elements.** The serialized attribute is `rowspan` (HTML standard). Tests query via `attributes('rowspan')` which returns the string value, or `undefined` if the attribute isn't on the element. `expect(...).toBeUndefined()` is the way to assert absence.

### Mechanisms understood (Phase 7)

### Reference reading (Phase 9)

### Mechanisms understood (Phase 9)

- **Row resolution belongs in the composable layer, not the core IR.** chronix's `BarDragTransaction` is a pure value type (kind + barId + originPx + deltaX + deltaY). Adding `projectedRowId` to it would push strip awareness into the DOM-free + layout-free `interaction/` module. Keeping it in the composable (`useGanttPointer`) where `strips` already live preserves the layer separation. The composable exposes `projectedRowId: ComputedRef<string | null>` for live read-out; the render path consumes it for snap geometry.
- **Half-open interval is the right model for strip ownership.** A pointer y exactly on `strip[i].y` belongs to strip[i] (lower-inclusive), and a y exactly on `strip[i].y + strip[i].height` belongs to NO strip when `rowSpacing > 0` (upper-exclusive). This mirrors chronix's other layout passes (axis tick ranges, viewport visible ranges) and matches the geometric reality of the rendered 1-px row-divider border.
- **Intra-strip offset preservation is what makes the snap feel right.** A bar 8 px below its source strip's top sits 8 px below the target strip's top mid-drag. Top-aligning to `targetStrip.y + barVerticalPadding` would look correct for uniform-height strips but would jump the bar visually when target strip height differs. Preserving the offset is also what the reference's mirror does. Implementation: `renderY = targetStrip.y + (bar.y - sourceStrip.y)`.
- **The 1-px rowSpacing gap is real geometry, not a special case.** When the pointer slides through the divider between two strips, `defaultStripResolver.atY` returns null and the render briefly free-falls by deltaY. This shows up in the snap-probe browser verify as a 1-stage rectY=70.5 between rectY=51 (snap-to-b) and rectY=94 (snap-to-c) — the bar is correctly "in transit" through the gap. Surprising but consistent with the catalog (item 19); a future phase could collapse the gap by treating divider y as belonging to the strip above OR below.

## Phase 10 — Theme tokens (2026-05-15)

### Reference reading (Phase 10)

- `packages/gantt/src/theme/GlobalThemeOverrides.ts` — ~25 partial-overridable tokens (`pageBgColor`, `eventBgColor`, `buttonHoverBgColor`, etc.). All distributed via CSS custom properties injected at the root, not via prop merging.
- `packages/gantt/src/theme/applyThemeOverrides.ts` (via imports) — runtime application path: iterate the partial map and set CSS variables on the gantt root element.
- `examples/gantt/vue3/src/DemoApp.vue:1313-1586` — demo's `themeOverrides` object passed as a top-level scheduler option.

### Mechanisms understood (Phase 10)

- **Flat namespace wins over nested groups for partial overrides.** Vue's prop-merging convention is `{...defaults, ...overrides}` — one level. Nested groups (`theme.header.cellFill`) would require either a custom recursive merge helper or noisy `:theme="{ ...defaults, header: { ...defaults.header, cellFill: 'red' } }"` consumer syntax. Flat with semantic-prefix names (`headerCellFill`) keeps both the type and the consumer call site clean.
- **TS-side merging is a superset of CSS-vars for chronix's purposes.** k-ui's CSS-var approach has the advantage of DevTools-tweakable values without re-running JS. The disadvantages: cross-framework chronix adapters (Vue3, React, Vue2 planned) would need parallel CSS distribution code, and TS-side merging is trivially type-safe whereas CSS-vars-as-strings lose type info at the boundary. Chronix's prop-merging cost is one re-render per theme change; CSS-vars save that round-trip but at the cost of a CSS-in-JS layer chronix has so far avoided.
- **Default theme values must be byte-equal to the prior inline literals** for the "zero-impact migration" promise to hold. Verified end-to-end by `chronix-verify` showing all 5 VRT baselines pass after the Commit 2 callsite swaps. If a single token's default had drifted by one hex digit, one of the 5 PNGs would have flagged it.

### Reference reading (Phase 11)

### Mechanisms understood (Phase 11)

- **Two distinct surfaces in k-ui: content replacement vs style overrides.** `eventContent` replaces the bar's inner DOM with consumer VNodes. `eventStyleCallbacks` overrides specific style properties (text color, bg color, border color) per event via lookup callbacks. - **Chronix bars are SVG `<rect>` leaf elements**, not HTML divs. Replacing the inner DOM doesn't apply — there IS no inner DOM. A custom bar renderer instead replaces the **entire** bar render: emit whatever VNode tree the consumer chooses (`<rect>`, `<g>`, `<path>`, multi-element compositions). - **Registry-based dispatch activates the dormant IR types.** Phase 1 designed `SlotContext / SlotTemplate / SlotRegistry / SlotRenderer` for exactly this pattern. Phase 11's API uses `createSlotRegistry()` + `slotRegistry?: SlotRegistry` prop, so the IR types finally have a real consumer. Alternative `barRenderer?: (ctx) => VNode` prop was considered and rejected — it doesn't scale to multi-slot APIs the way a registry does (future phases can register `'progress-overlay'`, `'tick-label'`, `'header-cell'` slots into the same registry).
- **Slot args carry live render geometry, not raw placedBar.** Mid-drag the bar's effective `renderX/Y/W/H` differs from `placedBar.x/y/width/height` because of `deltaX/deltaY` shifts and the Phase 9 cross-row snap math. ## Phase 12 — Selection model (`bar-click` + controlled selection) (2026-05-15)

### Reference reading (Phase 12)

- `packages/gantt/src/interactions/EventClicking.ts:1-71` — `eventClick` handler. Uses `listenBySelector('click', '.gantt-event', ...)` (DOM click event, not pointer-up). Drag-vs-click discrimination via `wasDragged` data attribute on the seg element. Suppresses click when target hits `gantt-just-dragged` / `gantt-event-progress-handle` / `gantt-event-progress-drag-triangle` classes.
- `packages/gantt/src/interaction/interactions/UnselectAuto.ts:1-72` — clicks outside any bar clear the selection via `dispatch({ type: 'UNSELECT_EVENT' })`. Configurable via `unselectAuto: boolean` + `unselectCancel: string` selector.
- `examples/gantt/vue3/src/DemoApp.vue:1681-1685` — demo's `handleEventClick` just confirms + deletes the bar. Minimal usage example.

### Mechanisms understood (Phase 12)

- **Pointerup vs DOM click event.** k-ui uses a separate `click` listener; chronix routes through the existing pointerup handler. Routing through pointerup means the drag-vs-click discrimination has one source of truth (the same handler that decides commit-vs-abort also decides emit-click-or-not), avoiding the k-ui-style data-attribute dance where the drag interaction marks an element and the click handler reads the mark later.
- **Zero-delta drag abort.** With `editable: true`, a plain click on a bar body would have started a `bar-drag` transaction at pointerdown. Phase 12's adapter detects 0-delta on the way out: `bar-drag` / `bar-resize` / `calendar-range-select` with no time/space change → abort instead of commit. This preserves the "click fires click, drag fires drop" intuition without polluting `onBarDrop` consumers with no-op events. Progress-handle transactions DON'T get the abort treatment — reaching the handle hit zone is itself the user's intent, so a 0-delta progress-handle pointerup still commits (with `resolvedProgress === initialProgress` — semantically a confirm-current).
- **lastHit preserved through commit, cleared by next begin.** Pre-Phase-12 `commit()` cleared `lastHitResult` (matching `abort()`'s discipline). Phase 12 needs lastHit AFTER commit so the click decision can read which bar was hit. Removed the clear from `commit()`; relied on `begin()`'s unconditional `lastHitResult.value = hit` to overwrite at the next pointerdown. `abort()` still clears (browser-initiated cancel discards everything by design).
- **Controlled selection over internal state.** chronix's pattern across editable / selectable / links / theme / slotRegistry is "consumer owns state, adapter is a function of props". A built-in selection ref would have created two sources of truth (the prop AND the internal Set). The `useGanttSelection()` composable provides ergonomic stateful wrapping without compromising the controlled-prop model.
- **`shiftKey` for multi-select.** macOS file managers use Cmd-click for toggle, Windows uses Ctrl-click. Web list UIs typically use Shift for range-select. None of these match perfectly. Phase 12 picks `shiftKey` for toggle because: (a) it's modifier-key consistent across OS, (b) range-select is parked anyway (catalog item 10), (c) future range-select could use shift+up/down arrows or shift+click-on-another-row pattern that doesn't conflict with toggle.

## Phase 14 — Resizable sidebar (single area divider) (2026-05-15)

### Reference reading (Phase 14)

- `packages/gantt/src/resource-timeline/ResourceTimelineLayout.tsx:5, 26, 61, 65, 74, 126-138, 199-204, 304-342` — the reference's single divider between resource panel and timeline. Width-override state (`null` until user drags), MIN/MAX clamp at 30 / `viewWidth - 30`. No emit on commit; override stays in component state.

### Mechanisms understood (Phase 14)

- **Single divider, not per-column.** The reference doesn't have a per-column divider anywhere. Per-column dividers would multiply state (one override per column) and add UX complexity (which column does "the" boundary belong to). The area-only model is cleaner and matches the reference exactly.
- **Override state, not prop.** Putting the override in component state matches the reference's behavior — prop-driven width is the INITIAL, drag-driven width is the live. A prop would either become controlled (consumer has to wire `:sidebar-width-override + @sidebar-resize`) or fight the drag handler. Keeping it internal stays out of the consumer's way; if a consumer later needs persistence, that's a follow-up phase with a small additive surface.
- **Proportional scaling preserves col ratios.** Chronix's `ColumnSpec.width` carries user intent ("region is 60, base is 100, name is 80 → 3:5:4 ratio"). Scaling each by the same multiplier (`effective / sumOfBase`) preserves the ratio at every override width. "Last-column-absorbs-delta" (the alternative) would have made the rightmost column variable-width and the others permanent-width, which breaks the ratio intent.
- **`<colgroup>` reuse is a Vue patching footgun.** A single `colGroup` VNode embedded in two parent trees triggers Vue's "same vnode reference" short-circuit on patch — only one of the two DOM locations receives prop updates after a state change. The bug was latent pre-Phase 14 because the col widths never changed reactively (they were fixed at `ColumnSpec.width`). The proportional-scaling test exposed it. Fix: replace `const colGroup = h(...)` with `const buildColGroup = () => h(...)` called once per use site.

### Mechanisms understood (Phase 16)

- **The reference's `isDragging` gate is what makes "click vs drag" clean.** Without it, every pointerdown on a draggable element would fire `eventDragStart` even if the user immediately released without moving. With it, a click-without-movement fires neither start nor stop. Chronix's mirror: a `dragStartFired: Ref<boolean>` latch in `useGanttPointer`, reset in `begin()`, set to true on the first `advance()` with non-zero delta. Stop callbacks check the latch before firing.
- **Lazy-start solves chronix's "begin immediately" architecture.** Chronix's pointer composable starts a transaction at pointerdown (not after a distance threshold) because the transaction layer is pure-functional and needs an anchor immediately. Firing start eagerly at `begin()` would emit dragstart even on aborted 0-delta clicks, which would break consumers tracking "is something being dragged right now". Lazy-start sidesteps the gate-vs-state-machine mismatch by deferring the emit until the first non-zero advance.
- **`commit()` order: stop → commit-callback.** Reference's `handleDragEnd` line 450 (eventDragStop) precedes line 457 (mutation merge + eventDrop). Chronix mirrors: in `commit()`, fire stop BEFORE the kind-specific commit branch. This lets consumers reset "drag in progress" UI state before the mutation arrives — important when the mutation handler does something expensive (e.g. server save) and you want the spinner to fall away promptly.
- **`abort()` fires stop too.** Reference fires `eventDragStop` regardless of whether the mutation was valid; the only difference is whether `eventDrop` follows. Chronix's abort path (pointercancel + 0-delta-after-non-zero-advance edge case) needs to fire stop symmetrically so consumer drag-state resets cleanly. Gated by the same `dragStartFired` latch — an aborted transaction that never advanced doesn't fire a stop with no matching start.
- **Adapter is the right place to enrich the payload.** The composable callbacks carry just `{ barId }` (+ `edge`) because `useGanttPointer` is event-source-agnostic and doesn't take `bars` as input. The adapter has `props.bars` and the in-flight PointerEvent, so attaching `sourceBar` + `jsEvent` at the emit site is a one-line `find` + ref read — keeps the composable narrow and matches the existing `bar-click` enrichment pattern from Phase 12.
- **Refresh `lastJsEvent` at pointerup AND pointercancel.** The composable's `commit()` / `abort()` fire stop callbacks synchronously from the pointerup / pointercancel handlers. If `lastJsEvent` were last set at pointermove time, the stop emit would reference the wrong event (the move event, not the up/cancel event). Refreshing at pointerup / pointercancel makes the stop payload reference the event that actually ended the gesture.

### Mechanisms understood (Phase 19)

- **Same-row overlap is rendering-correct, not just k-ui convention.** Chronix's `BarStackHeightPass` greedy-packs same-row bars into vertically stacked sub-rows. Two bars on the same row with intersecting time ranges therefore don't visually collide — they share the row height and stack. Cross-row overlap is the meaningful conflict signal (one bar leaves row A at hour 10 and another bar arrives on row B at hour 10 — a scheduling conflict the host might want to forbid). The `if (stillBar.rowId === proposal.rowId) continue` skip in `satisfiesOverlap` is therefore semantically aligned with `BarStackHeightPass`'s output, not just a tribute to the reference's `isSameResourceRow` shortcut.
- **Pure validators in `packages/gantt/src/api/validation.ts` keep the core's "pure pixel-and-time math" purity intact.** The adapter (vue3) does the wiring: it pulls validator props off the component, builds a `ValidationContext` (which resolves `MaybeRefOrGetter` to plain values), and dispatches to the core's `validateDrop` / `validateResize` / `validateSelect`. The session layer (`defaultPointerCaptureSession`) stays validation-free — its job is producing the `resolvedRange`, not gating it. Decision documented as alternative #3 in the design doc.
- **Rejection-emit shape is the host's main observability hook.** The `*-rejected` payloads carry `oldRange`, `attemptedRange`, and `reason`. A host can show `'cannot move past 8pm (constraint)'` toast text directly from `reason` without re-running the predicates. Three separate emits (rather than one with a `rejected: { reason }` field on existing emits) means `bar-drop` is exclusively about successful commits — host code branches on emit kind, not a flag.
- **Bar-stack visual stacking + same-row exclusion fits both v0 BarSpec semantics AND k-ui's resource-row semantics.** K-ui's `eventDef.resourceIds: string[]` can attach one event to MULTIPLE resources; chronix `BarSpec.rowId: string` is single. Chronix v0 doesn't multi-row by design (parked). The single-row exclusion matches the reference's effective behavior when `subject.resourceIds.length === 1`, which is the common case.
- **Progress-handle is the only commit that intentionally skips validation.** Reasoning: changing progress is a value mutation (number 0..100), not a date/row mutation. Validators take `DropProposal { range, rowId }` — there's no field for "progress changed". A future "progress validator" would be `progressAllow(barId, oldProgress, newProgress)` and live in a separate path. The unit test `progress-handle commit is never gated by eventConstraint` pins this regression-guard.
- **The validator gate is opt-in.** Each validator prop is optional; when `bars` isn't threaded into `useGanttPointer` (test fixtures that don't pass it) or when all 4 validator props are undefined (host hasn't wired any), the gate is a no-op.
