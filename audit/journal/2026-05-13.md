### Mechanisms understood (slot-width derivation)

- **k-ui doesn't hardcode per-view slot widths.** Slot width = `max(viewportWidth / slotCount, minLabelCell)`, where `minLabelCell = isTimeScale ? 4*13 : 5*13` (i.e. 52 for day/week clock-time labels, 65 for date labels like `15日三`). When the viewport is wide enough that one slot would naturally stretch past the floor, slots stretch; otherwise the floor holds and the timeline becomes scroll-wider than the viewport.
- **`isTimeScale` is per-view-type.** day/week → true (hour-based labels); month/season/halfYear/year → false (date-based labels). The chars-13 floor reflects fitting "≥4 / ≥5 Han chars + minor padding" without label truncation.
- **DOM probe confirms the formula at viewport 1440×900:** `.gantt-timeline-body-wrapper` width per view = day 1440, week 8736, month 2015, season 5980, halfYear 11960, year 23725 → divides cleanly by `slotCount` to give 60/52/65/65/65/65, which is exactly `max(1440/slotCount, floor)`. Empirically VIEWPORT.width drives the stretch term (matches k-ui's actualBodyWidth in this demo because the chart container is given full width).

### Mechanisms understood (bar placement)

- **Date math is local-TZ across both sides.** The demo's `formatDateTime` produces strings like `"2026-05-13 13:00:00"` (space-separated, no TZ); `new Date(str)` interprets these as local time on whatever runtime parses them. Under Playwright clock-install + system TZ shared between Node and the browser, both produce the same epoch — so a chronix `BarSpec.range` built from `today + N*24h + H*1h` (in ms) lines up exactly with k-ui's parsed event timestamps.
- **k-ui only renders events that intersect the day-view's date range.** Chronix's `BarPlacementPass` produces a placement for _every_ input bar regardless of axis range (negative x or `x + width > totalWidth` allowed); the render layer is what clips. Compare by `data-event-id` intersection — chronix-produced events absent from DOM are simply not in scope for visual parity.
- **k-ui's bar geometry is `(start - axisStart) × pxPerMs` and `(end - start) × pxPerMs`** — same formula chronix uses (`bar-placement-pass.ts:35-42`). With slot-width parity already locked in, this reduces to "do both sides see the same `start`/`end` Date instants?" — yes, when local-TZ + clock are shared.

### Mechanisms understood (Y placement)

- **Body has flat leaves, not a tree.** All 32 resources render as contiguous lanes in render order (which k-ui sorts by base/group internally — the demo's `RESOURCES[]` order is NOT the rendered order; first lane is `id=10`, not `id=32`). So Y parity doesn't need tree-aware grouping — just per-row heights and the correct order.
- **k-ui's bar Y formula is decoupled from strip height.** Bar lives at `strip.y + 8` with `height = eventMinHeight (30)`, regardless of strip.height. Strip.height varies 39–43px depending on stacking; the bar is fixed and centered-ish (asymmetric: 8px above, 1–5px below).
- **Wrapper has a ~0.5px internal offset** before the first lane starts, so the body's first row top relative to the wrapper is 0.5, not 0. Chronix's `RowSwimlaneLayout` starts strip[0].y at 0 by contract; parity must add the offset back.

When set, `bar.height = barHeight` and `bar.y = strip.y + barVerticalPadding` regardless of strip dimensions. When unset, falls back to the v0 symmetric formula `height = strip.height - 2 × padding`. v0 unit tests untouched; new vitest case covers explicit barHeight + asymmetric semantics.
