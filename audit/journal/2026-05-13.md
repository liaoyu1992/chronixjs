### Mechanisms understood (slot-width derivation)

- **k-ui doesn't hardcode per-view slot widths.** Slot width = `max(viewportWidth / slotCount, minLabelCell)`, where `minLabelCell = isTimeScale ? 4*13 : 5*13` (i.e. 52 for day/week clock-time labels, 65 for date labels like `15日三`). When the viewport is wide enough that one slot would naturally stretch past the floor, slots stretch; otherwise the floor holds and the timeline becomes scroll-wider than the viewport.
- **`isTimeScale` is per-view-type.** day/week → true (hour-based labels); month/season/halfYear/year → false (date-based labels). The chars-13 floor reflects fitting "≥4 / ≥5 Han chars + minor padding" without label truncation.
- **DOM probe confirms the formula at viewport 1440×900:** `.gantt-timeline-body-wrapper` width per view = day 1440, week 8736, month 2015, season 5980, halfYear 11960, year 23725 → divides cleanly by `slotCount` to give 60/52/65/65/65/65, which is exactly `max(1440/slotCount, floor)`. Empirically VIEWPORT.width drives the stretch term (matches k-ui's actualBodyWidth in this demo because the chart container is given full width).

## Stack-aware row height pass (BarStackHeightPass)

### Reference files read (stacking)

- `packages/gantt/src/resource-timeline/GanttView.tsx:2046-2148` (upstream reference) — `calculateRowHeights(rowNodes, eventMinHeight, eventSpacing, splitProps, tDateProfile, timelineWidth): number[]`. Computes per-row height via per-row slicing + `computeFgSegPlacements` + `firstEventTopPadding + maxBottom + 4`, floored at `minRowHeight = eventMinHeight + 8`.
- `packages/gantt/src/timeline/event-placement.ts:44-146` — `computeFgSegPlacements` returns `[placements, totalHeight]`; each placement has a `top` (level-coord, multiples of `height + spacing`). Stacking uses an interval-graph `SegHierarchy` for strict-order + max-stack capping.
- `packages/gantt/src/resource-timeline/GanttView.tsx:2348-2366` — `resolveFirstEventTopPadding(option, resourceId)` resolves a global number / per-resource map / default 8. Demo sets `resourceFirstEventTopPadding: 8` globally (DemoApp.vue:1576).

### Mechanisms understood (stacking)

- **The upstream reference uses an interval-graph stacking algorithm** for max-stack capping and strict-order. For chronix v1, simple greedy interval coloring is sufficient (assign each sort-by-start bar to the lowest level whose latest bar ended before this one starts). Both yield the same `maxLevel` for any input where no bars are dropped — and the demo doesn't use max-stack capping, so the simpler algorithm is exact-equivalent for the parity dataset.
- **Boundary touches (`bar1.end === bar2.start`) share a level.** The upstream's hierarchy uses non-strict less-than for end ≤ start. Chronix mirrors via `levelEnds[l] <= bar.start`.
- **Out-of-axis bars don't count toward stacking.** k-ui's `slicer.sliceProps(...)` clips events to the active date range before placement; an event whose range falls entirely outside the rendered axis doesn't push its row taller.

### Chronix work derived (stacking)

## VirtualizedPaneLayout (Phase 2 pass #5 — closes Phase 2)

### Reference files read (virtualization)

- None this iteration. ### Mechanisms understood (virtualization)

- **Layout passes output positions in the FULL coordinate space.** Bars, links, ticks all carry absolute (x, y) into the timeline body's content area, not viewport-relative. - **Overscan** is the layout's contribution to scroll smoothness: pre-render a few rows / columns just outside the visible window so the next scroll frame already has them. v0 accepts `{ rows, slots }` overscan separately.

## Phase 3 entry — PointerCaptureSession + BarDrag

### Reference reading (Phase 3 entry)

- No new references this iteration. Phase 1 IR design already pre-baked the `requireInitialHit` flag on `PointerCaptureConfig` and the four transaction discriminants — Phase 3 just adds the runtime that produces and mutates those values.

### Mechanisms understood (Phase 3 entry)

- **Chronix's interaction layer is adapter-agnostic.** The session has no DOM bindings, no event listeners, no pointer-capture API calls — adapters (Vue3/React) do that. The session exposes a pure-functional reducer trio (begin/advance/commit) over an immutable transaction value.
- **`requireInitialHit` is the entire "separate-SVG-layer" escape valve.** Caller decides whether the pointerdown hit the subject element (hit testing is the caller's job — DOM-specific) and passes a boolean; combined with the per-subject config, the session returns either a transaction or `null`. This single config field replaces the entire "progress-triangle hit obscured by overlapping layers" empirical workaround in the upstream reference.
- **Advance is cumulative-from-origin, NOT cumulative-from-previous.** This makes the reducer idempotent over a pointer-move trace — replay the same `currentPx` sequence and you get the same final transaction. The transaction carries `originPx` so advance is stateless.
- **Snap-to-grid rounds the time delta, not the pixel delta.** Pixel delta is converted to ms first; the snap is applied in ms. This avoids accumulating rounding error from view-zoom changes (snap at the time level, render at the pixel level).

### Chronix work derived (Phase 3 entry)

- `packages/gantt/src/interaction/pointer-capture-session.ts` (rewritten — was types-only stub) — defines the `PointerCaptureSession` interface and `defaultPointerCaptureSession` impl with `beginBarDrag` / `advanceBarDrag` / `commitBarDrag`. Plus input/output types: `BeginBarDragInput`, `AdvanceBarDragInput`, `CommitBarDragInput`, `CommitBarDragOutput`, `PointerPx`.
- `packages/gantt/src/interaction/transactions.ts` — `BarDragTransaction` gained `readonly originPx: { x, y }` so advance() recomputes deltas without caller bookkeeping. Other transaction kinds untouched.
- 17 vitest cases — see commit `6b857d2` description for the full enumeration. Notable corners: snap rounding at exact midpoint (Math.round vs banker math); `snapDurationMs ≤ 0` treated as no-snap; commit doesn't mutate input range.

### Naming justifications (Phase 3 entry)

- `defaultPointerCaptureSession` is an object literal with methods, not a class. Stateless — multiple drags can run in parallel (one per touch-id) by holding their own transaction objects.
- Input types `BeginBarDragInput` / `AdvanceBarDragInput` are kind-specific (one per transaction discriminant). When BarResize / ProgressHandle / CalendarRangeSelect arrive, they'll add their own begin/advance/commit triplets to keep each transaction's API typed-narrow.

## ProgressHandle transaction runtime (Phase 3)

### Reference reading (ProgressHandle)

- `packages/gantt/src/interaction/interactions/EventProgressing.ts:130-140, 207-209` — the upstream's progress-drag math:

  ```ts
  // begin (pointerdown):
  initialProgress = progress.value;
  initialX = mouseEvent.clientX;
  eventWidth = eventRect.width; // event seg's getBoundingClientRect

  // advance (dragmove):
  deltaX = mouseEvent.clientX - initialX;
  progressDelta = (deltaX / eventWidth) * 100;
  newProgress = clamp(0, initialProgress + progressDelta, 100);
  ```

  The upstream also throttles updates by 0.5% / 16ms (lines 211-222), but that's render-throughput throttling — the final commit-time progress is the unthrottled formula. Chronix's session models the unthrottled math; any throttling is the adapter's concern.

### Mechanisms understood (ProgressHandle)

- **Progress-drag is RELATIVE.** Final progress = initialProgress + (pointer delta / barWidth × 100). NOT absolute (pointer-x → fraction of bar). This means a 60-pixel drag on a 6060-pixel bar shifts progress by 1.0%, regardless of where on the bar the drag started.
- **`requireInitialHit: false` is the canonical config.** The progress triangle renders in a separate SVG overlay layer (`gantt-progress-triangles` group with pointer-events:auto). The chart's primary hit-test pass doesn't index that band, so `pointerdown` on the triangle wouldn't pass a `requireInitial=true` gate. The upstream sets `hitDragging.requireInitial = false` (EventProgressing.ts:79); chronix's config field is the typed analog.
- **Mid-drag values may go out of [0, 100].** Caller / renderer's choice whether to display clamped or unclamped. Commit clamps unconditionally.

### Chronix work derived (ProgressHandle)

- `transactions.ts` — `ProgressHandleTransaction` extended with `readonly originPx: PointerPx`, `readonly initialProgress: number`, `readonly barWidth: number`. Same shape pattern as the BarDrag extension.
- `pointer-capture-session.ts` — three new input/output types (`BeginProgressHandleInput`, `AdvanceProgressHandleInput`, `CommitProgressHandleInput/Output`) plus three method impls. Zero-`barWidth` guard so advance doesn't divide by zero (returns `initialProgress` unchanged).
- 11 new vitest covering: begin pinning + `requireInitialHit` semantics; right/left drag math; out-of-range projected values mid-drag; idempotent advance (cumulative-from-origin); zero-barWidth no-op; commit clamping both ends; in-range passthrough; **and a recording-replay case that lifts pointer-down x=1241.79, pointer-up x=1301.79 from `recordings/progress-handle-drag/log.json`, runs chronix's begin/advance/commit with `initialProgress=50`, `barWidth=6060`, and asserts `Math.round(resolvedProgress) === 51`** — matching the recorded after-state event text `A33-液压系统检修 - 51% 进行中`. First Phase 3 parity assertion.

### Naming justifications (ProgressHandle)

- Methods named `beginProgressHandle` / `advanceProgressHandle` / `commitProgressHandle` (vs e.g. `progress.begin`). ## CalendarRangeSelect transaction runtime (Phase 3 core complete)

### Reference reading (CalendarRangeSelect)

- No new upstream reading. Range-select is a fundamentally different interaction from drag/resize: there's no "original range" to displace; both ends come from the drag itself. Chronix's choice to work in time-space at this boundary (rather than pixel-space, like the other three kinds) is a chronix-native design call, not a port.

### Mechanisms understood (CalendarRangeSelect)

- **Time-space at the boundary, not pixel-space.** The caller has already converted the `pointerdown` pixel position to an `anchorTime: Date` via the axis. Reason: there's no anchor "original" to preserve a pixel offset against; the entire range is defined by the drag, so storing it directly as a `(anchorTime, currentTime)` Date-pair is more natural than pixel-then-convert.
- **Snap semantics diverge.** `BarDrag`/`BarResize` snap the **delta** (preserves the relationship to original range). `CalendarRangeSelect` snaps **each end independently** to a snap-grid multiple. The two semantics correspond to "shift this thing by a snap-grid amount" vs "land both ends on grid lines from scratch."
- **Snap may collapse to zero.** Both ends within ±half-snap-duration of the same grid mark collapse to zero-duration. Caller's policy whether that's a valid event create.
- **Backward drag auto-swaps.** If user drags from a later time to an earlier time (`currentTime < anchorTime`), commit produces `{ start: min, end: max }` so the resolved range always has `start <= end` regardless of drag direction.

### Chronix work derived (CalendarRangeSelect)

- `pointer-capture-session.ts` — `BeginCalendarRangeSelectInput`, `AdvanceCalendarRangeSelectInput`, `CommitCalendarRangeSelectInput/Output` types added; three new methods (`beginCalendarRangeSelect`, `advanceCalendarRangeSelect`, `commitCalendarRangeSelect`) on the session interface and impl. The stale "v0 implements `BarDragTransaction` only" docstring updated to declare all four kinds implemented and to highlight the time-space-vs-pixel-space asymmetry of this last one.
- 11 new vitest covering: begin (anchor pinning, requireInitialHit reject); advance (currentTime updates, anchor stays); commit forward / backward (auto-swap) / zero / snap rounds each end independently / snap collapse-to-zero / snap ≤ 0 treated as no-snap / rowId propagation; full begin/advance×N/commit lifecycle.

### Naming justifications (CalendarRangeSelect)

- `anchorTime` / `currentTime` (vs `originTime` / `cursorTime`). "Anchor" emphasizes the role: this is the point the drag is pivoting around; "current" is where the cursor is right now. Matches the existing IR field names from Phase 1 type design.
- Snap parameter is still `snapDurationMs` for cross-kind consistency, even though the math means "round each end to a grid multiple" not "round the delta". The two interpretations share the same unit and don't conflict at the call site.

### Open / parked (CalendarRangeSelect)

- **No parity test.** Would need a `select-to-create` recording from the upstream demo (`select` callback fires when range-select completes). Capture later in a recording batch.
- **Pixel↔time conversion belongs to the adapter, not the session.** That keeps the session adapter-agnostic. When the Vue3 adapter MVP arrives, it'll wrap the session with the conversion helper.

## select-to-create parity — analyzed and intentionally tautological

Tried to add a recording-replay parity vitest for `CalendarRangeSelectTransaction`. Investigated three capture strategies; concluded a strict parity assertion would be **tautological** with parity tests already in the suite.

### Capture strategies investigated

1. **Auto-dismiss the prompt + read the new bar from DOM.** Failed: the upstream's `select` callback fires, prompt opens, but with `dialog.dismiss()` no event is added. No new bar to read.
2. **Accept the prompt with a sentinel title + read the new bar.** Verified the demo's `addEvent` DOES fire (sentinel title appears in the sidebar's `currentEvents` list — `sentinelInDom=true`), but **the new bar never renders in day view** even after a 2s wait. Probable cause: the upstream's `selectInfo.allDay` resolves to true (or some view-filter quirk) when the drag happens in a time-grid view, so the added event doesn't render in the visible time band. Couldn't find an oracle to assert against.
3. **Intercept the upstream's `select` callback args.** Would require monkey-patching the scheduler options before mount via `page.addInitScript`, but the demo passes the callback directly into the Vue component — there's no global window hook to intercept cleanly without modifying the demo (R2 read-only).

### Why a parity assertion would be tautological

The chronix↔upstream parity for `CalendarRangeSelectTransaction` reduces to:

- chronix: `resolvedRange = { start: anchorTime, end: currentTime }` (or swapped)
- upstream: `resolvedRange = { start: timeFromPx(pointerDown.x), end: timeFromPx(pointerUp.x) }`

Both sides compute time via `axisStart + (pointer.x - wrapperLeft) / pxPerMs`. The constituent pieces are already pixel-perfect-verified by other parity tests:

- **`pxPerMs` parity**: covered by the 6 slot-width parity tests (day/week/month/season/halfYear/year) — chronix's `axis.slotWidth` matches the upstream's rendered slot spacing within 1px for every view.
- **`wrapperLeft` parity**: implicit in the bar-placement test, which uses chronix's `BarPlacementPass` output to predict bar pixel positions matching the upstream's rendered bars within 1px across 13 events. If `wrapperLeft` differed between chronix and the upstream, those 13 events would have systematic x offsets — they don't.
- **`axisStart` parity**: implicit in the day-view tick-labels set-equality test (chronix's `axis.ticks[0].time` matches the upstream's leftmost tick), and again in the bar-placement test.

So given chronix and the upstream agree on `pxPerMs`, `wrapperLeft`, and `axisStart`, they MUST produce the same `resolvedRange` from the same pointer events. A recording-replay assertion would just re-test those three components, not exercise any new code path.

### What stays committed

- `recordings/select-to-create/{before,mid,after}.png` + `log.json`: still useful as a smoke artifact (proves the demo's select-to-create flow works at the recorded time/view).
- The recording captures `wrapperLeft` alongside the pointer events, so a future parity vitest can be added cheaply if/when one becomes non-tautological (e.g. the `selectMirror` rectangle's bbox during drag — actually a snap-time decision the upstream makes, distinct from the math chronix already shares).
